## Google Antigravityでエージェントを"作る"という発想

[Tinkly](https://note.com/jake_k547)

こんにちは！

**Antigravity** 、入れてますか？  
私は入れてます。入れて...なんか色々独自の要素が多くて難しいので放置してました。

以前の記事で「ざっくり使った」って言ってから、正月休みもあって少し間が空いちゃったんですよね。で、久しぶりに触ろうと思ったんだけど、正直「 **コーディング以外に何に使えばいいんだ** 」状態になってまして。

今回はなんとか基本的な使い方を理解しつつ、その答えが見つかった話をします。

Antigravityの導入と使い方については以下の動画が分かりやすかったです。

## Antigravityを入れたけど、コーディング以外に使い道がない？

世の中のAntigravity記事や動画を見てると、だいたいこんな感じじゃないですか。

- 「Antigravityでバイブコーディング！」
- 「LPページを自動生成してみた」
- 「Gemini 3と一緒にアプリ開発」

いや、それはわかる。めっちゃ便利なのはわかる。

でも私、そもそもアプリ開発ってあんまりしないんですよ。興味がないわけじゃないんですが、作るとしても小規模のものか簡単なMCPサーバーくらい。

普段やってるのは **記事作成** とか、 **プロンプトの設計** とか、どちらかというとクリエイティブ寄りの作業。大規模なアプリを作るわけでもないし、Webサービスを開発するわけでもない。

じゃあAntigravityは私には合わないのか？

...と思ってたんですが、 **視点を変えたら一気に面白くなった** 。

## 「スキルを使ってエージェントを作る」という発想

### IDEじゃなくて「エージェントの住処」として見る

Antigravityを「IDE（統合開発環境）」として理解しようとすると迷う。

でも、 **エージェント運用の場所** として見ると、コーディング以外でもガッツリ使えることに気づいたんです。

ここで重要なのが「 **Skill-First** 」という考え方。

### Building（構築）から Equipping（装備）へ

従来の「エージェントを作る」って、こういうイメージでした。

![画像](https://assets.st-note.com/img/1768198676-S7DJtqFEYknXAMPfQxhjmCow.png?width=1200)

要するに、エージェントを「 **苦労して開発するもの** 」から「 **既存のLLMに道具を持たせて構成するもの** 」へと捉え直したわけです。

![画像](https://assets.st-note.com/img/1768198734-QYAKrGhiEudZa7J4oH56IcR2.png?width=1200)

この「道具」にあたるのが「 **スキル（Skills）** 」。

### スキルって何？

スキルは、エージェントに「新しい能力」を後付けでインストールするためのパッケージ。

実体は超シンプルで、 **フォルダ1つ** （\`SKILL.md\` + 補助スクリプト）だけ。

```python
my-skill/
├── SKILL.md        # スキルの説明・使い方
└── (補助ファイル)   # 必要に応じて
```

これを所定のフォルダに置くだけで、AIが「あ、自分はこの道具を使えるんだ」と認識して、必要なときに勝手に使い始める。

### スキルがエコシステムを超えて広がってきてる

ここが面白いところなんですが、このアンスロピックが発明したスキルという仕組み、 **Claude Code以外でも使える** んです。

![画像](https://assets.st-note.com/img/1768199113-XnQzxUdsKhCaPYqFMomp12LR.png?width=1200)

2025年12月にOpenAIもこの方式を採用。事実上の業界標準になってきてます。

これ、何がすごいかというと、 **Claude Code用に作ったスキルをCodexでも使える** ってこと。

で、今回私が勉強がてらやってみたのは、まさにこれ。

## 試しにAntigravityで動くプロンプト職人エージェントを作ってみた

### 何を作ったか

「 **PROMPTMAGIC** 」という名前のエージェント。

役割は、ユーザーの曖昧な依頼（「リサーチ用のプロンプト作って」みたいな）を受け取って、AIが処理しやすい **構造化されたプロンプト** に変換してくれる職人です。

いわゆる「プロンプトを作るためのエージェント」を手元に置けるのが面白い。

### 作り方が衝撃的だった

驚くほどシンプルでした。

1. **使ったツール**: OpenAI Codex
2. **持たせたスキル**: 「 **antigravity-agent-architect** 」（アンチグラビティのエージェント作成を支援するスキル）
3. **指示した内容**: 「プロンプトを整える職人エージェントが欲しい。 **antigravity-agent-architect** スキルを使って、必要なファイル一式を作って。」

必要なリソースを与えてこれだけ。

...いや、ホントにこれだけ。

Codexが自律的にディレクトリを作成し、設定ファイルを生成し、必要なルールやワークフローを配置してくれた。

![画像](https://assets.st-note.com/img/1768139118-786GqOZe43uaCPXz0n25UhdI.png?width=1200)

codexへの最初の指示画面

### PROMPTMAGICの3段階ワークフロー

今回お試しで作ったAntigravityのエージェント、いきなり答えを出さないんです。以下のステップを自律的に踏む設計になってる。

**Step 1: ヒアリング（お悩み聞き取り）**

- 不足情報を特定して質問する（最大3問まで）

**Step 2: 設計（プロの思考）**

- 「7つのコツ」に基づいてプロンプトの設計図を作る
- タスク分解、情報の順序、DO形式、短い例、役割設定など

**Step 3: 清書（仕上げ）**

- 設計図をもとに、コピペ専用の \`FINAL\_PROMPT.md\` を生成
- 余計な解説なし、プロンプト本文のみ
![画像](https://assets.st-note.com/img/1768139678-3Zs9TecA8WJanHOVY6b472pu.png?width=1200)

Antigravityのエージェントマネージャーの画面

### ちょっと面白い発見：「応援コメント」の効果

PROMPTMAGICを設計するとき、生成されるプロンプトの末尾に **人間っぽい一文** を含めるようにしてみたんです。

- 「よし、最高の仕事をしよう！」
- 「君ならできる！」

こういうやつ。

**意外と効果があった** 。

たとえば、同じプロンプトでも「君ならできる！」の一文があるほうが、出力の構造が整っていたり、細かい見落としが減る印象。科学的に証明できるわけじゃないけど、3回くらい比較して「あれ、違う？」と感じたんですよね。

無機質な命令文だけのプロンプトより、こうした「有能な同僚のような振る舞い」を含むコンテキストのほうが、モデルの実行精度と安定性が上がる感触がある。

まだ検証中なので断言はしないけど、方向性としては面白い。

## Repository as Agent - エージェントの「住処」という考え方

### Antigravity特有の概念

ここからがAntigravityならではの話。

Antigravityでは、 **リポジトリ構造そのものがエージェントの「身体」になる** という考え方があることに気づきました。

![画像](https://assets.st-note.com/img/1768199304-7urLp43cRefGH6SEIQKjz2bF.png?width=1200)

つまり、 **部屋を用意して家具を置いたら、勝手に住人が引っ越してくる** 感覚。リポジトリ1つがエージェントの「家」になるわけです。

![画像](https://assets.st-note.com/img/1768199626-Vr6OiuymhIe3oF9AGsQ4fEnH.png?width=1200)

### ファイル構成（PROMPTMAGICの場合）

```python
PROMPTMAGIC/
├── .agent/
│   ├── rules/
│   │   └── promptmagic.md              # 常時ルール（口調・制約）
│   └── workflows/
│       ├── promptmagic.md              # ワークフロー（全体の入口/統合）
│       ├── promptmagic-hearing.md      # Step 1: ヒアリング
│       ├── promptmagic-design.md       # Step 2: 設計
│       └── promptmagic-finalize.md     # Step 3: 清書（最終出力）
├── .gitignore
├── GEMINI.md                           # 目的・使い方・重要ルール
└── prompt_tips_compact.md              # 参照資料（7つのコツ）
```

### GEMINI.md / Rules / Workflows の役割分担

ここが地味に大事なポイント。

![画像](https://assets.st-note.com/img/1768199749-kl9hH0pLzFc7Sw3sfGvU58Xd.png?width=1200)

**ルールと手順を混ぜない** のが重要。

- ルール＝いつでも守るべき制約
- ワークフロー＝時系列の手順

これらが混同されるとエージェントが困ります。ここの設計が難しいところなんですよね。

実は今回のPROMPTMAGICについては、スキルの初期バージョンで作った後にAntigravity上で調整をしたものになってまして、若干ルールとワークフローなどの切り分けがうまくできていないんですが、とりあえず動作はします。。。　

でもこうやってエージェントを実際に使ってみながら調整する作業は楽しいですね。

### エージェント置き場リポを作った

今後、作成したエージェントは \`antigravity-agents/\` というリポジトリに集約していく予定。

```ruby
~/Developer/antigravity-agents/
├── PROMPTMAGIC/          # プロンプト職人
├── (次のエージェント)/
└── ...
```

単なるコード置き場じゃなくて、それぞれが人格（\`GEMINI.md\`）を持ったエージェントの実体が並ぶ場所。

この構造自体が「My AI Team」を可視化する面白いポイントだと思ってます。育てていくのが楽しみ。次はまたスキルを使って「リサーチ職人」とか作りたいなと（笑）。MCPサーバーと連携したり、ツールを装備させるのもいいですよね。

![画像](https://assets.st-note.com/img/1768141688-qD2h6HIijPenR1VxOwXNJp5K.png?width=1200)

エージェント置き場の画面

## 「Claude Codeで全部やればいいじゃん？」問題

ここまで読んで、こう思った人もいるかもしれません。

> いや、それ  
> **最初から全部Claude Codeでやればよくない？**

……うん、正論です（笑）。  
実際、 **技術的にはその通り** 。PROMPTMAGICみたいなエージェントも、Claude Code単体で作れるしやれます。

でも、Antigravityを使うのには理由があります。

### 理由①：Claude Codeの「使用量」を節約できる

まず現実的な話。

Claude Codeって、便利だけど **作業を全部任せると消耗が激しい** んですよね。 **コンテキストも使用量もガリガリ削られる** 。

そこで、

- **Antigravity / Codex CLI** ：  
	👉 エージェントの骨組み作り・構造生成・試作・情報整理
- **Claude Code** ：  
	👉 仕上げ・記事作成・最終調整・表現の磨き込み

みたいに **仕事を分業** すると、Claude Codeが節約できるんですよ。  
更に、AntigravityからClaude Codeが呼び出せるので連携もしやすい。

### 理由②：エージェント視点だと、Antigravityも楽しい

これは完全に主観なんですが……  
**正直、楽しい** 。

特にいいのが、エージェントを動かしたときに表示される **Artifacts** 。

- 途中経過が視覚的に見える
- 「今、何を考えて、どこまで進んだか」が分かる
- なんか…… **かっこいい**

Claude Codeのログとは違って、  
**エージェントが“働いてる感”を眺められる** のが好きなんですよね。

![画像](https://assets.st-note.com/img/1768196177-7CbloOyVf0QDYtjIrBvkS3Em.png?width=1200)

遊びで作った冷蔵庫の救世主エージェントのアーティファクト画面。 こんなタスクでもしっかりプランニングしてやってくれます。

### 結論：合理性＋趣味です

まとめると、

- Claude Codeだけでもできる
- でも分業したほうが使用量も気持ちも軽い
- 何より、 **そのほうが楽しい**

これが今回Antigravityを使った一番の理由です。

ツールは「正解」を選ぶものじゃなくて、  
**自分が気持ちよく続けられる使い方** を見つけるもの。

今回は、その一例として  
「Antigravityでエージェントを作る」という選択をしてみました。

## おまけ：配布物の紹介

この記事を読んで「自分も試してみたい」と思った方向けに、2つ配布します。

### おまけ①：antigravity-agent-architect（スキル）

Antigravityのエージェント雛形を作るためのスキル。

- ルール / ワークフロー周りを「ゼロから作る」のが面倒だったので作った
- **雛形作成 → 調整** の前提で使うのがコツ
- 私の嗜好に合わせて作ってあるので、万人向けではない
- 完璧ではないし、現在も調整を続けています（今はバージョン3.1です）

ご興味がありましたらスキルに対応しているAIツールにセットして使ってみて下さい。

クロードデスクトップの場合はzipファイルのままアップロードすればそのまま使えますが、完成したファイルが個別で出力されることがあるため、その場合は「zipファイルにして」と指示して下さい。

完成したエージェントの中身は念の為確認したほうがいいです。

特に中間、最終成果物はデフォルトではアーティファクトのディレクトリに保存するように設定しているので、このパスが書かれていた場合は間違っていることがあるため消すか、抽象化するか、保存場所を指定する必要があります。

Antigravityは自分のことをよく理解しているので、このリポジトリのエージェント何か問題ない？ みたいに聞いてみるのもいいと思います。

### おまけ②：PROMPTMAGIC（エージェント）

今回の主役「プロンプト職人エージェント」の試作品。

- 解凍してAntigravityで開くだけ
- 「リサーチ用」「記事構成用」「要約用」など、用途別にプロンプトを整えるのに便利

### 使い方

1. ZIPをダウンロードして解凍
2. 適当なディレクトリに配置
3. Antigravityでそのフォルダを開く
4. 動く

それだけです。

## まとめ：自分に合ったツールを選ぶ時代

今回わかったこと。

**Antigravityは「IDE」として見ると迷うけど、「エージェントの住処」として見ると一気に面白くなる。**

- スキルを使えば、コードを書かなくてもエージェントが作れる
- そのスキルは、Claude Code / Codex CLI / Gemini CLI間で使い回せる
- アンチグラビティの場合: ルール、ワークフロー、GEMINI.mdを配置したリポジトリを用意して開くだけで、エージェントが住み始める

この手のツールって本当にたくさん出てきてますよね。Claude Code、Cursor、Windsurf、そしてAntigravity。

どれが「最強」というよりかは、 **自分の用途に合ったツールを選ぶ時代** になってきてるんだと思います。

私の場合：

- 記事作成 → Claude Code（サブエージェント・機能が豊富）
- Claude Codeを補佐するエージェント作成・実験 → Antigravity または Codex
- 伏兵 → Agent Zero

勿論ChatGPTとかも沢山使ってますが、こんな使い分けに落ち着きつつあります。

皆さんも、自分なりの「AI活用スタイル」を見つけてみてください。
