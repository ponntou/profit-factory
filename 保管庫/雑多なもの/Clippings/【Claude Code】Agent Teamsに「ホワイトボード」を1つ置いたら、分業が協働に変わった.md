## 「ホワイトボード」を1つ置いたら、分業が協働に変わった

Claude CodeのAgent Teamsで、同じ分析タスクを2回実行しました。

1回目はデフォルト構成で2人のエージェントにそれぞれ別の役割を割り当てて並列実行しました。結果は **2つの質の高い独立レポート** でした。しかし2つを並べると、互いの内容を参照した記述がどこにもありませんでした。

2回目は **リーダーが共有Markdownファイル（ホワイトボード）を1つ用意しただけ** です。結果はこうなりました。

| 観点 | 1回目 - デフォルト構成 | 2回目 - ホワイトボードあり |
| --- | --- | --- |
| **相互参照** | なし | Agent BがAgent Aの発見を明示的に参照・検証 |
| **横断的洞察** | 0件 | 4件の横断的観察が出現 |
| **用語の統一** | 各自が独自の用語を使用 | Agent Aの用語をBが引き継ぐ |
| **検証行動** | なし | Agent Aの予測をBが4件中4件検証 |
| **成果物の性質** | **2つの独立レポート** | **1つの統合分析** |

SendMessage（Agent Teams標準のメッセージ機能）は会話であり、ホワイトボードは共有知識です。 **ホワイトボードがチームワークを促進するのは、人間もAIも同じようです** 。

## 実験の詳細: 何が起きたか

### タスク設計

**目標**: Next.js App Routerのアーキテクチャベストプラクティスを分析する  
**チーム構成**: Agent A（ルーティング・データフェッチング担当）+ Agent B（パフォーマンス最適化担当）

|  | Case 1: デフォルト構成 | Case 2: ホワイトボードあり |
| --- | --- | --- |
| **構造** | Agent A, B（独立して並列実行） | リーダーがホワイトボード作成 → Agent A実行・書き込み → Agent B実行 |
| **共有情報** | なし | ホワイトボードに書かれた発見事項 |
| **実行順序** | 並列 | 逐次（Aの知見をBが参照） |

### Case 1（デフォルト構成）: それぞれ完全、でも互いを知らない

Agent Aはルーティングとデータフェッチングについて網羅的に分析しました。Server Componentsでの `fetch` 、 `generateStaticParams` 、Parallel Routes、Intercepting Routesなど。

Agent Bはパフォーマンスについて詳細にまとめました。画像最適化、 `next/font` 、Route Segment Config、Partial Prerendering（PPR）の設定方法など。

**問題**: 2つのレポートは **それぞれ完全ですが、互いを参照していません** 。Agent AはServer Componentでの `fetch` を分析しましたが、キャッシュ戦略との関係には触れていません。Agent Bはキャッシュの設定方法を分析しましたが、どのデータフェッチングパターンと組み合わせるべきかには言及していません。

### Case 2（ホワイトボードあり）: 予測し、参照し、検証する

Agent Aがホワイトボードに書き込んだ中に、こんなセクションが出現しました。

```
### Agent Bへの申し送り（パフォーマンスに影響しそうな設計判断）
- Next.js 15以降、fetch はデフォルトで非キャッシュ → 明示的に \`cache: 'force-cache'\` の指定が必要
- fetch は layout/page/generateMetadata 間で自動的に重複排除される（Request Memoization）
- generateStaticParams でプリレンダリングされるページ → ビルド時にキャッシュ済み
- Parallel Routes (@folder) は独立してストリーミング可能 → loading.tsx の配置が鍵
```

Agent Aは **自分の担当範囲を超えて、Agent Bの仕事に役立ちそうな予測を書き残していました** 。

Agent Bはこのホワイトボードを読んでから分析を開始しました。その結果、Case 1にはなかったセクションが出現しました。

```
### Agent Aのデータフェッチング設計 → パフォーマンスへの影響
- Next.js 15のデフォルト非キャッシュ → 意図的に \`force-cache\` や \`revalidate\` を
  設定しないとSSR毎にfetchが走る。revalidate 値がTTFBに直結
- fetch の Request Memoization → 同一レンダリングパス内で同じURL+オプションの
  fetchは自動的に1回にまとめられる。ただしORM等は React cache() で明示的にメモ化が必要
- Parallel Routes + Suspense → 各 @slot が独立ストリーミング可能。
  最も遅いslotがページ全体のLCPを決める

### Agent Aが予測した設計判断 — パフォーマンス検証結果
- fetch デフォルト非キャッシュ → ✅ 確認。\`use cache\` ディレクティブで明示的にキャッシュ可能
- fetch の重複排除 → ✅ 確認。layout/page/generateMetadata間で機能する
- generateStaticParams → ✅ ビルド時キャッシュ確認。dynamicParams: false で未生成ページを404化可能
- Parallel Routes + streaming → ✅ 確認。ただし error.tsx の配置で個別エラー境界が必要
```

さらに、Case 1には存在しなかった **「横断的な観察」セクション** が生まれました。

```
## Cross-Cutting Observations
1. fetch のキャッシュ戦略は「データフェッチングの設計」と「パフォーマンス」の
   両方の文脈で理解する必要がある — 片方だけでは設計判断を誤る
2. layout.tsx は「ルーティング構造」であると同時に「キャッシュ境界」でもある
3. Suspense boundary の配置は、ルーティング設計とCore Web Vitalsの
   両方を考慮して決める必要がある
4. Route Segment Config (dynamic, revalidate) は
   データフェッチングの設計とパフォーマンスチューニングの接点
```

## ホワイトボードパターン: 5分で試せるテンプレート

ホワイトボードパターンは、 **チームメンバーが読み書きできる共有Markdownファイル** を置くアプローチです。Agent Teamsの標準機能ではなく、リーダーのプロンプトに組み込むパターンです。

```
リーダーがホワイトボード作成 → Agent A 実行・書き込み → Agent B が読んでから実行
```

今回の実験ではA → Bの逐次実行でしたが、B → Aの順序や、両者が並列でホワイトボードを共有する構成も可能です。

以下のテンプレートをリーダーのプロンプトにそのまま組み込めます。

```
# Whiteboard: {トピック}

## Goal
{チームが最終的に答えるべき問い}

## Team Structure
- Agent A: {担当領域}
- Agent B: {担当領域}

## How Our Work Connects
{2人の仕事の接続点を明示}

## Key Questions
1. {両方の視点が必要な問い}
2. {両方の視点が必要な問い}
3. {両方の視点が必要な問い}

---

## Agent A Findings
（Agent A: ここに書く）

## Agent B Findings
（Agent B: ここに書く）

## Cross-Cutting Observations
（どちらでも: 横断的な観察を書く）
```

ポイントは3つです。

1. **ゴールの明示** — 何のために分析するのか
2. **接続点の明示** — 2人の仕事がどうつながるか
3. **書き込み欄の構造化** — 各エージェントが書く場所と、横断的な観察を書く共有スペース

## なぜこれで変わるのか

### Agent Teamsのデフォルト: 実は共有されていない

Agent Teamsでチームメイト間で共有されるものと、されないものを整理します。

|  | 共有される？ | 仕組み |
| --- | --- | --- |
| **会話履歴** | ❌ されない | 各エージェントは独立したコンテキストウィンドウ |
| **発見事項** | ❌ 自動では共有されない | SendMessageで明示的に伝える必要あり |
| **タスクリスト** | ✅ 共有される | 全員がTaskList/TaskUpdateで読み書き可能 |
| **ファイルシステム** | ⚠️ 読み書き可能 | チームメイトの権限設定によっては書き込みが制限される |

[公式ドキュメント](https://code.claude.com/docs/en/agent-teams) にも、各エージェントは独立したコンテキストウィンドウを持ち、会話履歴は共有されないと明記されています。連携手段はSendMessage（メッセージング）とタスクリストに限られます。

つまり標準の情報共有手段は **SendMessageとタスクリスト** だけです。会話履歴もリーダーの知識も、チームメイトには引き継がれません。

### 人間のチームでも同じことが起きている

考えてみると、これは人間のチームでもよくある話です。

Slackでメッセージを送り合っているだけでは、チームは「分業」にとどまります。でも会議室のホワイトボードにゴールと進捗を書き出した瞬間、メンバーの動きが変わる——そんな経験はないでしょうか。

チーム研究の分野では、これを **「共有メンタルモデル」** と呼んでいます。チームメンバーが同じ全体像を持つことで、明示的な指示がなくても互いの行動を予測し、調整できるようになるという考え方です。

ホワイトボードが果たしている役割を整理すると、こうなります。

| やっていること | 効果 |
| --- | --- |
| **ゴールと接続点を書く** | チーム全体の目的と、自分の仕事が他のメンバーにどう影響するかが見える |
| **ファイルを全員に共有する** | 異なる専門領域の間を橋渡しする共有物になる |
| **チーム構成を明示する** | 「誰が何を担当しているか」が分かり、相手の仕事を意識した行動が生まれる |

つまりホワイトボードは、チャットでは伝わらない **「全体像」と「つながり」** を可視化しています。

### 実は40年前からあるアイデア

このアプローチには先行研究もあります。1986年にNiiが提唱した **[Blackboard Architecture](https://ojs.aaai.org/aimagazine/index.php/aimagazine/article/view/537)** は、複数のエージェントが共有知識ベースに読み書きして協調するアーキテクチャです。

2025年の研究（ [Han & Zhang](https://arxiv.org/html/2507.01701v1) ）では、LLMマルチエージェントシステムにこのアーキテクチャを適用した結果、 **静的なマルチエージェントシステムと同等以上の性能** を達成しつつ、 **トークン消費量も大幅に削減** できたと報告されています。

Agent Teamsでの「ホワイトボード」は、このBlackboard Architectureの軽量な実装と見ることができそうです。

## 実践: 使い分けと注意点

### いつ使うか

| 実行方式 | 適したタスク |
| --- | --- |
| **並列（ホワイトボードなし）** | 完全に独立したタスク |
| **並列（ホワイトボードあり）** | ゴールの共有だけで十分なケース |
| **逐次（ホワイトボードあり）** | 先のエージェントの発見が後の作業に影響するケース |

### いつ使わないか

ホワイトボードが常に最善とは限りません。

- **完全に独立したタスク** — 別々のリポジトリへの変更、共有コンテキストがない作業。ホワイトボードを置いても読む理由がなく、オーバーヘッドになるだけです
- **単純な並列実行** — 「テストを書く」と「ドキュメントを書く」のように接点が少ないタスク。SendMessageで十分です
- **3人以上のチーム** — ホワイトボードの構造が複雑になり、かえって混乱する場合があります。この場合は役割ごとにセクションを分けるか、複数のホワイトボードに分割する工夫が必要です

### リーダーの役割変化

ホワイトボードパターンでは、リーダーの役割が変わります。チームメイトはファイルへの書き込みが権限で制限される場合があるため、 **リーダーがチームメイトの発見をホワイトボードに転記する** 必要があります。

これは一見すると手間ですが、人間のチームでのリーダーと同じ役割です。会議でホワイトボードに要点を整理するファシリテーター。Slackで重要な議論をまとめて投稿するマネージャー。やっていることは同じだと感じています。 **チームの知識を可視化し、構造化し、必要な人に届ける** 。

リーダーの仕事は「作業の指示」ではなく「知識のファシリテーション」になります。

### なぜファイルなのか

ホワイトボードの実装にはいくつかの選択肢があります。

| 方式 | 構造化 | 永続性 | チームメイトが書ける？ |
| --- | --- | --- | --- |
| **Markdownファイル** | ✅ | ✅ ディスクに残る | ⚠️ リーダーが更新 |
| **SendMessageリレー** | ❌ テキストのみ | ❌ 会話に埋もれる | — |
| **タスクリスト** | △ タイトル+説明 | △ チーム解散で削除 | ✅ 全員可能 |

ファイルの最大の強みは **永続性** です。会話が終わった後もファイルは残ります。ホワイトボードは「コーディネーションツール」であると同時に「チームの成果物」でもあります。この二重の役割がファイルベースの強みです。

## まとめ

Agent Teamsで役割を分けるだけでは「分業」にしかなりません。ホワイトボードを置くことで「協働」に変わります。

1. **ゴールと接続点を共有する** — 各エージェントが全体像を意識できる
2. **発見事項を蓄積する** — 先のエージェントの知見が次の出発点になる
3. **横断的な洞察が生まれる** — 独立作業では出てこない、領域をまたいだ気づきが出現する

ホワイトボードはAgent Teamsの標準機能ではなく、Markdownファイル1つで実現できるシンプルなパターンです。次にAgent Teamsを使うとき、リーダーのプロンプトに「ホワイトボード」を組み込んでみるのもよいかもしれません。
