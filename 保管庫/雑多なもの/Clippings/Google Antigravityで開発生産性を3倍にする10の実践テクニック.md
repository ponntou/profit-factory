## はじめに

AI 支援開発ツールの進化により、エンジニアの働き方が大きく変わりつつあります。  
その中でも、Google Antigravity は強力なコード生成・編集機能を持つ AI エージェントとして注目を集めています。

しかし、多くの開発者が基本的な機能しか使っていないのが現状です。  
この記事では、Antigravity の真価を引き出し、開発生産性を劇的に向上させる実践的なテクニックを紹介します。

### この記事で学べること

- Antigravity の強力だが見落とされがちな機能
- 開発ワークフローを劇的に改善する具体的なテクニック
- JavaScript/TypeScript プロジェクトでの実践的な活用パターン
- AI エージェントとの効果的な対話方法

### 想定読者

- Antigravity を使い始めたが、まだ基本機能しか使っていない方
- AI 支援開発ツールをより効果的に活用したい中級エンジニア
- JavaScript/TypeScript 開発の生産性を向上させたい方

### 前提知識

- VSCode の基本的な操作
- JavaScript または TypeScript の基礎知識
- Git/GitHub の基本的な理解

## 基盤となる考え方

テクニックの紹介に入る前に、Antigravity を効果的に使うための基本的な考え方を理解しておきましょう。

### AI エージェントとの協働とは

Antigravity は単なるコード補完ツールではなく、 **自律的に判断して行動できる AI エージェント** です。  
「このコードを書いて」という指示だけでなく、「この機能を実装したい」という目的を伝えることで、最適なアプローチを提案してくれます。

### 効果的なプロンプトの原則

良いプロンプトの条件:

1. **目的を明確に**: 何を実現したいのかを具体的に伝える
2. **文脈を提供**: 関連するファイルや既存の実装を示す
3. **段階的に進める**: 大きな変更は小さなステップに分割
4. **日本語で自然に**: 専門用語も交えて普通に会話する

## 実践テクニック 10 選

それでは、実際に開発生産性を向上させる具体的なテクニックを見ていきましょう。

### テクニック 1: カスタムワークフローで反復作業を自動化

毎回同じような手順を踏む作業はありませんか？  
例えば、「新しいコンポーネントを作成 → テストファイルを追加 → Storybook ファイルを作成」といった定型作業です。

Antigravity の **ワークフロー機能** を使えば、こうした反復作業を自動化できます。

#### ワークフローファイルの作成

ワークフローは `.agent/workflows/` ディレクトリに配置する Markdown ファイルです。

.agent/workflows/create-component.md

```
---
description: Reactコンポーネントとテストファイルを作成
---

# Reactコンポーネント作成ワークフロー

このワークフローは、新しいReactコンポーネントと関連ファイルを一括作成します。

## ステップ1: コンポーネント名の確認

ユーザーにコンポーネント名を確認します。
PascalCaseで入力してもらいます。

## ステップ2: コンポーネントファイルの作成

// turbo

\`src/components/{ComponentName}/{ComponentName}.tsx\` を作成

- TypeScriptで型定義を含める
- Propsインターフェースを定義
- エクスポート文を含める

## ステップ3: テストファイルの作成

// turbo

\`src/components/{ComponentName}/{ComponentName}.test.tsx\` を作成

- React Testing Libraryを使用
- 基本的なレンダリングテストを含める

## ステップ4: index.tsの作成

// turbo

\`src/components/{ComponentName}/index.ts\` を作成し、コンポーネントを再エクスポート
```

#### ワークフローの実行方法

作成したワークフローは、ファイル名をスラッシュコマンドとして呼び出せます:

```
/create-component
```

Antigravity がワークフローファイル A 読み込み、ステップに従って作業を進めてくれます。

#### 実際の活用例

私が作成した **Zenn 記事作成ワークフロー** では、以下の手順を自動化しています:

1. ヒアリング（テーマ、ターゲット読者、目的）
2. アウトライン作成と承認
3. 記事ファイルの生成（ `npx zenn new:article` ）
4. Front Matter 設定
5. セクションごとの執筆
6. プレビュー確認
7. 公開準備

これにより、記事作成の標準的なフローが確立され、品質の一貫性が保たれています。

### テクニック 2: Rules ファイルで一貫した開発スタイルを維持

プロジェクトごとに「こういう風にコードを書いてほしい」というルールがありますよね。  
Antigravity の **Rules ファイル** を使えば、そうしたプロジェクト固有のルールを AI に学習させられます。

#### Rules ファイルの配置

ルールは `.agent/rules/` ディレクトリに Markdown ファイルとして配置します。

.agent/rules/coding-standards.md

```
# コーディング規約

## TypeScript型定義

- すべての関数に戻り値の型を明示する
- \`any\`型の使用は禁止（やむを得ない場合は\`unknown\`を使用）
- インターフェースよりも\`type\`を優先

## エラーハンドリング

- 非同期処理は必ずtry-catchでラップ
- エラーログには文脈情報を含める
- カスタムエラークラスを使用

## コメント

- 「何を」よりも「なぜ」を説明
- 複雑なロジックには日本語でコメント
- TODOコメントにはissue番号を記載

## ファイル構成

\\`\`\`
src/
  ├── components/    # UIコンポーネント
  ├── hooks/         # カスタムフック
  ├── utils/         # ユーティリティ関数
  ├── types/         # 型定義
  └── services/      # APIクライアントなど
\\`\`\`
```

#### ルールの適用範囲指定

ルールファイルのフロントマターで、適用範囲を指定できます:

.agent/rules/api-client.md

```
---
applies_to:
  - "src/services/**/*.ts"
  - "src/api/**/*.ts"
---

# API クライアント実装ルール

- すべてのAPIリクエストは\`apiClient\`を経由
- レスポンスの型を必ず定義
- リトライロジックを含める
```

このルールは `src/services/` と `src/api/` 配下のファイルにのみ適用されます。

#### 実践的な使い方

例えば、私の Zenn 記事作成プロジェクトでは、以下のようなルールを設定しています:

.agent/rules/create-article.md

```
# 記事執筆の品質基準

## 内容

- 導入部で記事の目的と学べることが明確
- ターゲット読者が明示されている
- すべてのコード例が動作する
- 適切な見出し階層（h2、h3を活用）

## スタイル

- 丁寧語（です・ます調）で統一
- 段落は3-5行程度で区切る
- Zennの拡張記法で補足情報を強調

## SEO

- タイトルは具体的で60文字以内
- トピックは適切（最大5つ）
```

こうすることで、Antigravity が記事を執筆する際、自動的にこれらの基準を満たすように動作します。

### テクニック 3: ナレッジファイルで文脈を共有

開発プロジェクトでは、「過去にこういう決定をした」「このアプローチは試して失敗した」といった文脈情報が重要です。

**ナレッジファイル** を使えば、そうした情報を Antigravity に参照させることができます。

#### ナレッジファイルの配置

ナレッジは `.agent/knowledge/` または `knowledge/` ディレクトリに配置します。

knowledge/architecture-decisions.md

```
# アーキテクチャ決定記録

## 状態管理ライブラリの選定（2026-01-01）

### 決定事項

Zustandを採用（ReduxではなくZustand）

### 理由

- TypeScriptとの親和性が高い
- ボイラープレートが少ない
- サイズが小さい（3KB）
- Reactのフックと自然に統合できる

### 却下した選択肢

- **Redux Toolkit**: チームの学習コストが高い
- **Recoil**: まだ実験的なステータス
- **Context API**: パフォーマンスの懸念

### 参考リンク

- [Zustand公式](https://github.com/pmndrs/zustand)
```

#### ルールファイルから参照する

Rules ファイルで、特定のナレッジファイルを参照するよう指示できます:

.agent/rules/state-management.md

```
# 状態管理ルール

**参照ナレッジ**: \`@knowledge/architecture-decisions.md\`

- 状態管理にはZustandを使用
- Storeは機能ごとに分割
- アクションは非同期処理を含められる

## Store作成の例

\\`\`\`typescript
import { create } from 'zustand';

interface UserStore {
  user: User | null;
  setUser: (user: User) => void;
  clearUser: () => void;
}

export const useUserStore = create<UserStore>((set) => ({
  user: null,
  setUser: (user) => set({ user }),
  clearUser: () => set({ user: null }),
}));
\\`\`\`
```

#### ナレッジファイルの活用パターン

以下のような情報を記録しておくと効果的です:

1. **設計決定の記録**
	- なぜその技術を選んだのか
	- 何を検討したのか
	- トレードオフは何か
2. **トラブルシューティングメモ**
	- 過去に遭遇したエラーと解決方法
	- ハマりポイントと回避策
3. **ベストプラクティス集**
	- プロジェクト固有の実装パターン
	- チームで合意したコーディングスタイル
4. **API ドキュメント**
	- 社内 API の仕様
	- 外部サービスとの連携方法
実際のナレッジファイル例

私の Zenn プロジェクトでは、以下のナレッジファイルを用意しています:

- `workflow.md`: 記事作成の具体的な手順
- `zenn-article-writer.md`: 執筆のベストプラクティス
- `zenn-markdown-reference.md`: Zenn 独自の記法リファレンス
- `article-template.md`: 記事テンプレート

これらを組み合わせることで、一貫性のある高品質な記事を効率的に作成できています。

### テクニック 4: 複数ファイルの一括編集を効率化

大規模なリファクタリングで、同じ変更を複数のファイルに適用する必要があることがよくあります。  
Antigravity の **multi\_replace\_file\_content** 機能を使えば、1 つのファイル内の複数箇所を効率的に編集できます。

#### 使い方のポイント

通常の `replace_file_content` は 1 つの連続したブロックしか編集できませんが、  
`multi_replace_file_content` を使えば、 **非連続な複数箇所** を一度に編集できます。

例えば、以下のような変更を一度に行えます:

```
// 変更前
import { OldComponent } from "./old";

export const Feature1 = () => {
  return <OldComponent data={data} />;
};

// ... 100行のコード ...

export const Feature2 = () => {
  return <OldComponent config={config} />;
};
```

これを新しいコンポーネントに一括で置き換える場合:

```
このファイル内のOldComponentをすべてNewComponentに変更して
```

と指示するだけで、Antigravity が:

1. import 文の変更
2. Feature1 内のコンポーネント置き換え
3. Feature2 内のコンポーネント置き換え

をすべて一度に実行してくれます。

#### 実践例: API エンドポイントの一括変更

「API\_V1 を API\_V2 に変更して、すべてのエンドポイントを更新して」と指示すれば、  
3 つの関数すべてが一度に更新されます。

### テクニック 5: ブラウザサブエージェントで UI 開発を加速

Web 開発で「この UI を作って」と指示すると、Antigravity が自動的にブラウザを開いて動作確認をしてくれます。

これは **browser\_subagent** という機能で、ブラウザ操作を自動化できます。

#### どんなときに使えるか

以下のような作業が自動化できます:

1. **UI コンポーネントの動作確認**
	- 作成したコンポーネントをブラウザで開いて表示確認
	- レスポンシブデザインのチェック
2. **フォームの動作テスト**
	- 入力フィールドに値を入力
	- バリデーションエラーの表示確認
	- 送信処理の動作確認
3. **ページ遷移のテスト**
	- リンククリックでの遷移確認
	- 戻る・進むボタンの動作確認

#### 実践例

```
ログインフォームを作成して、実際にブラウザで動作確認してください
```

このように指示すると、Antigravity は:

1. ログインフォームコンポーネントを作成
2. 開発サーバーを起動（まだの場合）
3. ブラウザでページを開く
4. フォームに実際に入力してみる
5. 動作の様子をレポート

という一連の流れを自動で実行してくれます。

ブラウザ操作の様子は自動録画される

browser\_subagent が実行したブラウザ操作は、WebP 形式の動画として自動的に記録されます。  
これにより、どのような操作が行われたか後から確認できます。

#### UI デバッグでの活用

バグ修正の際も便利です:

```
ボタンをクリックしたときにモーダルが表示されない問題を調査してください
```

Antigravity が:

1. ブラウザを開いてボタンをクリック
2. 開発者ツールでエラーを確認
3. 問題のあるコードを特定
4. 修正案を提示

という流れで、デバッグ作業を支援してくれます。

### テクニック 6: ターミナルコマンドの安全な自動実行

開発中に何度も同じコマンドを実行するのは面倒ですよね。  
Antigravity の **SafeToAutoRun** 機能を理解すると、効率が大きく向上します。

#### SafeToAutoRun とは

通常、Antigravity がコマンドを実行する際は、ユーザーの承認が必要です。  
しかし、安全なコマンド（読み取り専用など）は自動実行させることができます。

```
# 自動実行される（安全）
npm ls
git status
cat package.json

# 承認が必要（破壊的な可能性）
npm install
git commit
rm -rf node_modules
```

#### ワークフローでの活用

ワークフローファイルで `// turbo` を使うと、そのステップは自動実行されます:

.agent/workflows/setup-project.md

```
## ステップ1: 依存関係の確認

// turbo

\\`\`\`bash
npm ls --depth=0
\\`\`\`

## ステップ2: ビルドの実行

\\`\`\`bash
npm run build
\\`\`\`
```

この例では、 `npm ls` は自動実行されますが、 `npm run build` はユーザーの承認を待ちます。

#### 実践的な使い方

開発フロー全体をスムーズにする例:

```
TypeScriptのビルドエラーを確認して、修正してください
```

と指示すると:

1. `npm run build` を自動実行（SafeToAutoRun が true に設定される）
2. エラーメッセージを解析
3. 該当ファイルを修正
4. 再度ビルドして確認

という流れで、承認待ちなしに作業が進みます。

#### コマンド実行の監視

長時間実行されるコマンドも、Antigravity がバックグラウンドで監視してくれます:

```
# 開発サーバーの起動
npm run dev
```

このようなコマンドは、起動後も Antigravity が状態を監視し、  
エラーが発生したら通知してくれます。

### テクニック 7: コードアウトラインで大規模コードベースを把握

初めて触るコードベースや、しばらく見ていなかったファイルを理解するのは大変です。  
**view\_file\_outline** 機能を使えば、ファイルの構造を素早く把握できます。

#### コードアウトラインとは

ファイル全体を読む前に、以下の情報を一覧表示してくれる機能です:

- クラスの一覧と行番号
- 関数の一覧とシグネチャ
- メソッドの一覧
- 各要素の行範囲

```
// UserService.ts の構造
UserService (class)
  ├── constructor (lines 5-10)
  ├── getUser (lines 12-25)
  ├── createUser (lines 27-45)
  ├── updateUser (lines 47-65)
  └── deleteUser (lines 67-80)

Helper functions
  ├── validateEmail (lines 85-90)
  └── hashPassword (lines 92-100)
```

#### 効率的なコード理解の流れ

1. **まずアウトラインで全体像を把握**
	```
	src/services/user-service.tsの構造を見せて
	```
2. **気になる部分だけを詳しく見る**
	```
	UserService.createUserメソッドの実装を見せて
	```
3. **関連する部分も確認**
	```
	validateEmail関数も見せて
	```

このように段階的に理解を深めることで、無駄な情報を読まずに済みます。

#### 大規模ファイルでの活用

1000 行を超えるような大きなファイルでも、アウトラインを見れば:

- どこに何が定義されているか
- どの関数がどれくらいの規模か（行数）
- ファイルの責務範囲

が一目で分かります。

実例: React コンポーネントの理解

```
Dashboard.tsxのアウトラインを表示して
```

すると:

このように、コンポーネントの構造が一目で分かります。

#### リファクタリング計画に活用

大規模なリファクタリングを行う前に、複数のファイルのアウトラインを比較することで:

- 重複したコードの発見
- ファイル分割のポイント特定
- 責務の整理

といった計画を立てやすくなります。

「このコンポーネントはどこで使われているか」「この API エンドポイントを呼んでいる箇所は」といった検索をする機会は多いですよね。

**grep\_search** を使えば、プロジェクト全体から効率的にコードを検索できます。

#### 基本的な使い方

```
TodoListコンポーネントを使っている箇所をすべて見つけて
```

Antigravity が:

1. プロジェクト全体を検索
2. 該当箇所をファイル名と行番号付きで表示
3. 各箇所のコード内容も提示

してくれます。

#### 正規表現での高度な検索

正規表現を使った複雑な検索も可能です:

```
fetch('/api/で始まるAPIコールをすべて検索して
```

このように指示すれば、特定のパターンに一致するコードを一括で見つけられます。

#### 実践例: 未使用の import 文を探す

```
// どこでも使われていないimport文を見つける
import { UnusedComponent } from "./components";
```

「UnusedComponent を使っている箇所を検索して」と指示し、  
結果が 0 件なら、その import は削除できます。

#### リファクタリング前の影響範囲調査

大きな変更を加える前に、影響範囲を把握するのに便利です:

```
useAuthフックを使っているコンポーネントをすべて教えて
```

この結果から:

- どのファイルを修正する必要があるか
- テストでカバーすべき範囲
- 破壊的変更の影響度

を事前に把握できます。

### テクニック 9: 段階的な実装で品質を保つ

大きな機能を一度に実装しようとすると、レビューが大変だったり、バグが混入しやすくなります。

**段階的なアプローチ** で Antigravity と協働することで、品質を保ちながら効率的に開発できます。

#### ステップバイステップの指示

❌ 悪い例:

```
ユーザー管理機能を全部実装して
```

✅ 良い例:

```
ステップ1: まず、Userの型定義を作成して
ステップ2: 次に、APIクライアントを実装して
ステップ3: 最後に、Reactコンポーネントを作成して
```

#### 小さなコミット単位で進める

各ステップが完了するたびに、Git コミットを作成することを Antigravity に指示できます:

```
型定義が完成したら、Gitにコミットしてください
```

これにより:

- 変更履歴が明確
- 問題があれば簡単に巻き戻せる
- レビューしやすいコミット単位

が実現できます。

#### フィードバックループを回す

```
まず基本的な実装をして、動作確認してください。
問題があれば修正して、次のステップに進んでください。
```

このように指示すると、Antigravity が:

1. 基本実装を作成
2. 自動で動作確認（ビルド、テスト実行）
3. エラーがあれば修正
4. 問題なければ次へ

という流れで進めてくれます。

#### 実践例: フォームコンポーネントの段階的実装

```
1. まず、フォームの型定義とバリデーションルールを作成
2. 次に、バリデーションロジックのテストを書いて
3. その後、基本的なUIコンポーネントを実装
4. 最後に、送信処理とエラーハンドリングを追加
```

各ステップで動作確認しながら進めることで、どこで問題が起きたかが明確になります。

実際の会話例

**あなた**: フォームの型定義を作成して

**Antigravity**: (型定義を作成)

**あなた**: ビルドエラーがないか確認して

**Antigravity**: (ビルド実行) 問題ありません。次のステップに進みますか?

**あなた**: はい、バリデーションのテストを書いて

このように、一つ一つ確認しながら進められます。

### テクニック 10: Artifacts 機能で成果物を管理

Antigravity が生成した画像やドキュメント、設定ファイルなどは、 **Artifacts** として自動的に保存されます。

この機能を理解すると、プロジェクト資産を効率的に管理できます。

#### どんなものが Artifacts になるか

1. **生成した画像**
	- UI モックアップ
	- アイコン、ロゴ
	- ダイアグラム
2. **ブラウザ操作の録画**
	- UI 動作確認の様子
	- バグ再現手順
3. **大きなドキュメント**
	- API 仕様書
	- アーキテクチャ設計書

#### 実践例: UI モックアップの生成

```
モダンなダッシュボードのUIを生成して
```

Antigravity が:

1. 画像を生成
2. Artifacts として保存
3. 画像をプロジェクトで使えるように配置

してくれます。

#### ブラウザ録画の活用

browser\_subagent が実行したブラウザ操作は、WebP 動画として Artifacts に保存されます。

これを活用すると:

- バグ報告に操作手順を添付
- UI 動作のドキュメント化
- チームメンバーへの共有

が簡単にできます。

#### 設計ドキュメントの管理

```
このプロジェクトのアーキテクチャ図を作成して
```

生成されたドキュメントは:

- Artifacts として保存
- バージョン管理できる
- 後から参照・更新できる

という利点があります。

## 実践例: TypeScript プロジェクトでの活用

ここまでのテクニックを組み合わせた、実践的な開発フローの例を見てみましょう。

### シナリオ: ToDo アプリにタグ機能を追加

#### フェーズ 1: 企画とルール設定

.agent/rules/todo-app.md

```
# ToDoアプリ開発ルール

- TypeScriptで型安全に実装
- すべてのコンポーネントにテストを含める
- Zustandで状態管理
- Zodでバリデーション
```

#### フェーズ 2: ワークフローの作成

.agent/workflows/add-feature.md

```
---
description: 新機能追加の標準フロー
---

## ステップ1: 型定義の作成
## ステップ2: APIロジックの実装
## ステップ3: UIコンポーネントの作成
## ステップ4: テストの追加
## ステップ5: 統合と動作確認
```

#### フェーズ 3: 実装（Antigravity との対話）

```
/add-feature でタグ機能を追加したい
```

Antigravity がワークフローに従って:

1. **型定義**: `Tag` interface を作成
2. **Store 拡張**: Zustand ストアに tags 配列を追加
3. **UI コンポーネント**: `TagInput` と `TagList` を作成
4. **テスト**: 各コンポーネントのテストを作成
5. **統合**: 既存の ToDo コンポーネントに組み込み
6. **動作確認**: ブラウザで実際に動かして確認

という流れで自動的に進めてくれます。

#### フェーズ 4: レビューと改善

```
TagInputコンポーネントをview_file_outlineで確認して
```

アウトラインを見て、気になる部分があれば:

```
handleSubmit関数の実装を見せて
バリデーションを強化してほしい
```

と段階的に改善していきます。

この例では、以下の作業が自動化されます:

- 型定義の作成（5 分 → 1 分）
- コンポーネントの基本実装（30 分 → 5 分）
- テストの雛形作成（15 分 → 3 分）
- 統合作業（10 分 → 2 分）

**合計: 60 分 → 11 分（約 5 倍の生産性向上）**

## よくあるアンチパターンと対策

Antigravity を使い始めた人が陥りがちな落とし穴と、その対策を紹介します。

### アンチパターン 1: 曖昧すぎる指示

❌ 悪い例:

```
良い感じにコンポーネントを作って
```

この指示では:

- 「良い感じ」の基準が不明確
- 何を実装すべきか分からない
- 結果が期待外れになる可能性が高い

✅ 改善例:

```
ユーザーのプロフィール情報を表示するカードコンポーネントを作成して。
- 名前、メールアドレス、アバター画像を表示
- レスポンシブデザイン（モバイル対応）
- TypeScriptの型定義を含める
```

### アンチパターン 2: 一度に大きすぎる変更を指示

❌ 悪い例:

```
認証機能、ユーザー管理、権限制御を全部実装して
```

問題点:

- レビューが困難
- どこでエラーが出たか特定しにくい
- 修正の手戻りが大きい

✅ 改善例:

```
まず認証機能だけを実装して。
ログイン・ログアウトの基本機能を動作確認してから、次のステップに進みたい。
```

### アンチパターン 3: ルールやナレッジを用意していない

❌ 悪い例:  
毎回、プロジェクトのコーディング規約を口頭で説明する

問題点:

- 説明が毎回微妙に変わる
- 一貫性が保てない
- 時間の無駄

✅ 改善例:  
Rules ファイルを作成して、一度記述すれば常に参照される

### アンチパターン 4: エラーが出たらすぐに諦める

❌ 悪い例:  
エラーが出たら「できませんでした」と報告だけ

✅ 改善例:

```
TypeScriptのビルドエラーが出ているみたいです。
エラーメッセージを解析して、修正してください。
```

Antigravity は自律的に問題を解決する能力があります。  
「調査して修正して」と明確に指示しましょう。

### アンチパターン 5: 文脈情報を提供しない

❌ 悪い例:

```
このAPIのレスポンス処理を追加して
```

問題点:

- どの API かが不明確
- 既存コードとの関係が分からない

✅ 改善例:

```
\`src/api/users.ts\` のfetchUser関数のレスポンス処理を追加して。
取得したユーザー情報をZustandのstoreに保存したい。
エラーの場合はトーストで通知する実装にしてください。
```

## まとめ

この記事では、Google Antigravity で開発生産性を大幅に向上させる 10 の実践テクニックを紹介しました。

### 10 のテクニックの振り返り

1. **カスタムワークフロー**: 反復作業を Markdown ファイルで自動化

単なる指示ではなく、対話を通じて:

- より良いアプローチを見つける
