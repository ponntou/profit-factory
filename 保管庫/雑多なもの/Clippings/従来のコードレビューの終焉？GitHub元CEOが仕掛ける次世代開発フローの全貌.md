## 本記事の背景

> "The concept of understanding and reviewing code is a dying paradigm."  
> （コードを理解しレビューするという概念は、死にゆくパラダイムだ）

2026年2月11日、Xに投稿されたこの一言が開発者コミュニティで話題になりました。  
投稿者は元GitHub CEOのThomas Dohmke（トーマス・ドムケ）氏で、2026年2月に新会社 [Entire](https://entire.io/) を設立し、6,000万ドル（約90億円）のシード資金を調達しています。

同時に、最初のプロダクトであるCheckpointsをリリースしたことも発表されました。  
これまで次々と新しいモデルや高機能なAIツールが登場してきたので、今回も新しいAIツールの登場かと最初は思いましたが、どうやら異なるようです。

本記事ではドムケ氏の主張とEntireの詳細を掘り下げつつ、コードレビューの終焉が意味するもの、そしてこれからの開発フローがどう変わるのかについて考察していきたいと思います。

## 「コード中心」という前提を疑う

先ほどのXの投稿内でも述べられているように、今回問題視されているのは **今の開発フローが「コード」を中心に回りすぎている** という点です。

> "We still depend on a software development lifecycle that makes code in files and folders the central artifact, in repositories and in pull requests."  
> （我々は依然として、ファイルやフォルダの中のコードを中心的な成果物とするソフトウェア開発ライフサイクルに依存している。リポジトリとプルリクエストの中で。）

Git・GitHub・プルリクエスト・コードレビューなどといった現在主流の仕組みは、当然ですが人間がコードを書き、人間がそれを読んで判断する前提で確立されてきました。

しかし、AIエージェントが大量のコードを生成し複数のエージェントが並行して作業する世界では、人間がコードを1行ずつ読んで理解するモデルはスケールしないということは明らかです。  
これが今回の「コードレビューは死にゆくパラダイム」という発言の背景にある問題意識だと思います。

確かに自分もAIエージェントを使った開発を進める中で、コードレビューによるボトルネックを感じることが多くなっています。  
いくら計画をきちんと立ててテストを充実させても、最終的にコードを人間が目視で確認するプロセスが残っていると、そこがどうしても遅くなります。

今後さらに開発フローへのAIの浸透が進み、AIが生成するコードが増えることで、この問題はさらに加速すると考えられます。

## Checkpoints（Entire CLI）によるAIとの協業

Entireの最初のプロダクトとしてCheckpoints（Entire CLI）がリリースされました。  
オープンソース（MITライセンス）で、Goで実装されています。

ひと言で言うなら、これは **AIエージェントとの開発セッション（プロンプト・対話・推論過程など）を、コード変更と一緒にGitへ記録するCLIツール** です。

### Checkpointsの特徴

従来のGitが記録するのはコードの変更差分だけです。

しかし、AIエージェントを使った開発では、変更差分だけでは情報が足りません。  
「なぜその変更が行われたのか」「どんなプロンプトで指示したのか」「AIはどんな推論過程を経てそのコードを生成したのか」が失われてしまいます。

そこで、Checkpointsは `Checkpoint` という単位で以下の情報をメタデータとして保存します。

- 人間とAIとの対話内容（トランスクリプト）
- 使用したプロンプト
- 参照・変更されたファイル
- 呼び出されたツール
- 消費されたトークン数

このメタデータは `entire/checkpoints/v1` という別ブランチにプッシュされるため、本体コードのブランチには一切影響を与えません。  
既存のGitワークフローに影響を与えることなく導入できるのは大きな利点ですね。

### 導入と使い方

インストールはHomebrewかGoで行います。

```
brew tap entireio/tap && brew install entireio/tap/entire
```

リポジトリで有効化するには `entire enable` を実行します。

```
entire enable
```

主要なコマンドは以下の通りです。詳細はREADMEを参照してください。

| コマンド | 内容 |
| --- | --- |
| `entire enable` | リポジトリでCheckpointsを有効化 |
| `entire status` | 現在のセッション情報を確認 |
| `entire rewind` | 前のCheckpointに戻す |
| `entire resume <branch>` | 過去のセッションを復元 |
| `entire disable` | フックを削除して無効化 |

### Checkpointの作成タイミング

Checkpointの作成タイミングには2つの戦略があります。  
`manual-commit` （デフォルト）はGitコミット時にCheckpointが作成され、 `auto-commit` はエージェントの各レスポンスの後にCheckpointが作成されます。

執筆時点ではClaude CodeとGemini CLIに対応しており、Codex、Cursor CLIなどへの対応も予定されています。

## Entireの主要3コンポーネント

さて、ここまでがEntireの現時点の話です。では「これから」はどうなるのか。  
Entire社は以下3つの主要コンポーネントを掲げており、上記で説明したCheckpointsはこれらの取り組みの第一歩と位置付けられています。

- Git互換データベース
- セマンティック推論レイヤー
- AI-nativeユーザーインターフェース

### ①Git互換データベース

コード、意図、制約、推論を統一的にバージョン管理することを目指す構想です。  
Checkpointsはそのビジョンに向けた最初の実装として、エージェントのコンテキストをGitに保存する仕組みを提供しています。

**従来のGitが「何が変わったか」を記録するのに対し、この層は「なぜ変わったか」「どういう文脈で変わったか」まで含めて記録** します。

> "And we begin shipping today with Checkpoints, a new primitive that automatically captures agent context as first-class, versioned data in Git."  
> （そして本日、Checkpointsをリリースします。エージェントのコンテキストを、Gitのファーストクラスデータとして自動記録する新しい仕組みです。）  
> *出典： [Xの投稿](https://x.com/ashtom/status/2021255786966708280)*

コードの変更履歴だけでなく、AIとの対話やプロンプト、トークン使用量までがGitのファーストクラスデータとして扱われます。

### ②セマンティック推論レイヤー

次に、セマンティック推論レイヤーというものが挙げられています。

> "Checkpoints is Entire's first move toward building a universal semantic reasoning layer for agents, according to Dohmeke. Today, it gives you traceability and history. Tomorrow, it will become the shared memory that allows agents to coordinate, hand off context and build together without collision or loss of understanding."  
> （Checkpointsは、エージェントのための普遍的なセマンティック推論レイヤーを構築するためのEntireの最初の一手だとドムケ氏は言います。今日、それはトレーサビリティと履歴を提供する。明日、それはエージェントが連携し、コンテキストを引き継ぎ、衝突や理解の喪失なしに共に構築するための共有メモリとなる。）  
> *出典： [Ex-GitHub CEO Backs a New Kind of Dev Platform for Human-Agent Collaboration](https://arc.net/l/quote/vwelhgpb)*

つまり、Checkpointsで蓄積されたコンテキストデータを使って、複数のAIエージェントが互いの作業内容を把握しながら協調して動けるようにするためのレイヤーを構想しているようです。

エージェント間で文脈を共有し、作業を協調させるための基盤になることが想定されています。  
具体的な実装や挙動については、現時点では詳細は公開されていません。

### ③AI-nativeユーザーインターフェース

こちらはおそらくまだ構想段階で詳細な情報が出ていませんが、人間がAIエージェントの「フリート（群れ）」を管理し、大量の変更をレビュー・承認・デプロイするためのインターフェースのようです。

- 人間のエンジニア＋多数のAIエージェント
- 1日に数百の変更が発生

このような世界を前提にしたインターフェースを構想しているようで、現在のような個別のプルリクエストでコード変更を1行ずつ確認するようなUIからは大きく変わるのではないかと思います。

## 開発フローはどう変わるのか

Entireのビジョンを踏まえると、これからの開発フローは以下のように変わっていくのではないかと考えています。

| これまで | これから |
| --- | --- |
| コード中心のSDLC（開発ライフサイクル） | エージェント時代に最適化されたSDLC |
| 人間同士の協業を前提としたプラットフォーム | 人間とAIエージェントが協調するプラットフォーム |
| リポジトリとPRが中心的成果物 | エージェントの活動・文脈も扱う基盤 |
| 個別の作業単位を人間が実装 | エージェントに作業を委任し、人間は監督・判断に集中 |
| 既存のSDLCを前提に改善 | エージェント時代に向けてSDLC自体を再設計 |

自分はAI駆動開発について「AIを道具から設計対象へ」というメンタルモデルの転換が必要だと以前記事でも書きましたが、Entireが提示しているのはさらにその先の世界に感じました。

## コードレビューは本当に「終わる」のか

では本当にコードレビューは終わるのかという点についてですが、  
自分はコードレビューの形が変わるだけでその本質は残ると考えています。

なぜならコードレビューの本質は「品質の担保」と「知識の共有」であり、これらはAIエージェントがコードを生成する世界においても重要なプロセスであり続けるからです。

ただし、スケールしないのは「差分を人間が読んで理解する」という従来の手段の方です。  
だからこそ、目的（品質の担保・知識の共有）は保ったまま、レビューの対象や手段を置き換える必要があります。

「品質の担保」という点では、AIが生成したコードを人間が目視で確認するのは非効率なので、  
テストや静的解析、CI/CDパイプラインなどの自動化された品質チェックがさらに重要になり、人間はそれらの結果をレビューして判断する形にシフトしていくのではないかと思います。

ただ、品質チェックを自動化に寄せたとしても、「知識の共有」という課題は残ります。  
特にAIが生成したコードの背景にある「なぜこのコードが必要なのか」「どんな意図でこのコードが書かれたのか」という情報は、コードだけでは伝わりにくいものです。  
従来このような情報はコードレビューの過程で共有されてきました。

EntireのCheckpointsはこの問題に対して、AIとの対話や推論過程をGitに記録することで、コードだけでなく「意図や推論」のレベルで知識を共有できるようにするアプローチを取っています。

つまり、コードレビューのプロセス自体がなくなるのではなく、レビュー対象の比重が「コード」そのものから「自動化された検証結果」や「意図や推論（プロンプト、制約、判断理由）」に移っていくのではないかと考えています。

## 最後に

AIエージェントがメインにコードを生成する世界において、人間の役割はコードを書いてレビューすることから、意図を設計しエージェントを監督することへとシフトしていくと考えられます。  
Entireはそのシフトを支えるプラットフォームを構築しようとしており、Checkpointsはその第一歩です。

まずはCheckpointsを実際に触ってみて、AIエージェントとの対話がGitに刻まれる世界がどんな開発体験になるのかを体感してみたいと思います。
