## はじめに

LLM がコードを生成する時代になりました。Claude Code や Codex のようなツールが日常的に使われるようになり、「コードを書く」という行為そのものが変わりつつあります。

この変化の中で、ソフトウェアアーキテクチャに求められるものも変わってきたのではないか、と感じています。

クリーンアーキテクチャや DDD は優れた設計思想です。ただ、従来のアプローチではレイヤーごとにディレクトリを分けることが多く、1つの機能を理解するために controller、service、repository、domain と複数の場所を見て回る必要があります。人間にとっても負担ですが、LLM にとってはなおさらです。コンテキストウィンドウに収める情報が増えるほど、焦点がぼやけやすくなります。

もちろん、大規模なシステムでは従来のレイヤー分けが必要になる場面はあります。ここで提案するのは、コンテキストを小さく保つことを軸に据えた、別のアプローチを考えてみようという試みです。

## 3つの基本方針

この記事で提案するアーキテクチャは、3つの方針に基づいています。

1つ目は、コンテキストの最小化です。ある機能を修正するとき、見なければならない範囲をできるだけ小さくします。これは LLM にとっても人間にとっても重要です。LLM は入力できるコンテキストに限りがありますし、人間も一度に把握できる範囲には限界があります。

2つ目は、純粋関数の最大化です。副作用のないロジックをできるだけ多くします。純粋関数はテストが簡単で、LLM が生成しやすく、人間にとっても理解しやすいという特徴があります。

3つ目は、Package by Feature です。機能単位でコードをまとめます。技術的な役割（controller、model、service）ではなく、業務上の機能（注文、配送、決済）を軸にファイルを配置します。レイヤー別の構成だと、1つの機能のために controller/、service/、repository/ と複数のディレクトリから関連ファイルを拾い集める必要があります。Package by Feature なら `/order/create/` をまとめて LLM に渡すだけで、その機能の全体像が伝わります。機能の追加や削除もディレクトリ単位になるので、影響範囲が明確です。

## ディレクトリ構造

Package by Feature に基づくディレクトリ構造の例を示します。

最上位にある `common` は業務に依存しない共通要素を置く場所です。金額や日時のような汎用的な型に加えて、テーブルに対応するモデルもここに置きます。このモデルは DB のスキーマと対応するだけで、ロジックは含みません。

各機能（order、shipping など）の中にある `shared` は、その機能内で共通して使うものを置く場所です。ここに置くエンティティは、テーブルのモデルから必要なフィールドだけを取り出して定義します。こうすることで、各機能は自分の関心事だけに集中できます。

参照ルールはシンプルです。 `common` はどこからでも参照できます。 `shared` はその機能の中でだけ参照できます。各機能の具体的な処理（create、cancel など）は外から参照しません。

この構造の利点は、ある機能を修正するときに見る範囲が明確になることです。「注文の作成」を直すなら `/order/create/` の中を見ればいい。共通のモデルに変更があれば `common` を見る。探す場所に迷いません。

## 純粋と非純粋の分離

ディレクトリ構造の中に `*.pure.ts` 、 `*.io.ts` 、 `*.usecase.ts` というファイル名が登場しました。これは純粋なロジックと副作用のある処理を分離するための命名規則です。

`*.pure.ts` は純粋関数を置くファイルです。外部の状態に依存せず、同じ入力には常に同じ出力を返します。注文金額の計算、割引率の適用、バリデーションなど、ビジネスロジックの多くはここに書けます。テストも入力と出力を検証するだけで済みます。DB の状態を気にする必要がないので、LLM がテストを繰り返し実行しながらコードを改善していくような使い方にも向いています。

`*.io.ts` は副作用のある処理を置くファイルです。データベースへの読み書き、外部 API の呼び出しなど、外の世界とやり取りする処理がここに入ります。

`*.usecase.ts` は両者を組み合わせるオーケストレーション層です。クリーンアーキテクチャでいう application 層に相当します。トランザクションの境界もここで管理します。典型的な流れはこうなります。

```
usecase:
  1. io でデータを取得
  2. pure でビジネスロジックを実行
  3. io で結果を保存
```

この分離が効くのは、「どこを見ればいいか」が一目でわかる点です。ロジックのバグなら `*.pure.ts` を見る。データの取得がおかしいなら `*.io.ts` を見る。流れ全体がおかしいなら `*.usecase.ts` を見る。LLM に修正を頼むときも、渡すファイルが明確になります。

## usecase と外部との接続

usecase がビジネスロジックの入り口だとすると、その外側にはフレームワークとの接続があります。Web フレームワークのコントローラーやルーティングがそれです。

フレームワークによっては、コントローラーの配置場所が決まっていることがあります。たとえば Next.js なら `app/` ディレクトリ、Rails なら `controllers/` ディレクトリに置く必要があります。

方針としては、フレームワークが指定する場所にはフレームワークが求めるファイルを置き、そこから usecase を呼び出すだけにします。

```
/app/api/order/route.ts       # フレームワーク指定の場所
  ↓ 呼び出すだけ
/features/order/create/create-order.usecase.ts
```

コントローラーは薄く保ちます。リクエストの受け取りとレスポンスの返却だけを担当し、ビジネスロジックには触れません。こうすることで、フレームワークへの依存はコントローラー層に閉じ込められます。フレームワークを変更しても、 `/features/` の中は影響を受けません。

## まとめ

この記事では、Package by Feature、純粋と非純粋の分離、命名規則による構造の明示という3つの要素を組み合わせたアーキテクチャを提案しました。

根底にあるのは「コンテキストの最小化」と「純粋関数の最大化」です。  
ある機能を修正するときに見る範囲を小さくし、ファイルの役割を名前から読み取れるようにする。  
さらに、純粋関数でDBなどの状態を切り離し、試行錯誤のサイクルを速くする。  
こうした構造は、LLM のための設計であると同時に、開発者自身にとっても見通しのよい設計です。

過剰な抽象化を避けつつ、ファイル構造と命名規則で安全網を張る。小さく、見通しよく、わかりやすく。この方針が設計の出発点になれば幸いです。

## 追記：人間の認知をボトルネックにしない

記事を公開した後、ある気づきがありました。

AI がコードを生成する時代、コードの量は爆発的に増えていきます。一方で、人間が把握できる範囲には限界があります。  
「全部のコードを隅々まで把握する」というスタイルはもう成り立ちません。

ここで pure / io の分離が効いてきます。

純粋関数は、入力と出力の関係だけ見ればいい。中身がどう実装されているかは、テストが通っていれば気にする必要がありません。  
人間がレビューすべきはセキュリティやパフォーマンスに関わる部分、つまり io の領域です。

認知リソースを io に集中させる。pure はテストに任せる。  
この分離によって、コードベースが大きくなっても人間の認知がボトルネックになりにくくなります。
