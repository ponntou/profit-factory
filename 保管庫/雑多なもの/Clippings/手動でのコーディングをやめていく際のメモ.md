Clineが出たあたりから最近に至るまで人間とcoding agentで協業しながらコードを書くというスタイルを取っていました。  
しかし、圧倒的な能力を持つClaude Opus 4.6の登場でいよいよコードを書くというエンジニアのアイデンティティのひとつを手放すときが来たかなと思い、2週間ほど前にダイナミックに実装のスタイルを変更しました。

その際に考えたことや経過などのメモです。

## 前提

- Claude Code
- ほぼTypeScript
- 中規模のサービス

## 最初に決めたこと

コードを書くのを止める、という方針を決めたときに2つの方針を作りました。

### あらゆる業務をドキュメント化する

LLMが人間と比較して明確に劣っている領域として、コンテキストの短さがあります。これは人間とやりとりをしながら長期間に渡るタスクを遂行する上で「タイミングによって実装方針が違う」「人間が同じ説明を何度もしなければいけない」といった不都合を生み出します。

自分は「coding agentがこの情報を利用するか？」という判断軸を捨てて、業務にかかわるあらゆる情報を `/docs` に保管して常に参照できるようにしていました。  
また、Claude Codeの場合 `/projects` にすべてのセッションが保管されているので、そこに対してPythonスクリプトでクエリを行うスキルを作って、coding agentが適宜呼び出せるようにすると良いです。

### 境界線を決める

人間とcoding agentで役割の境界線を引きました。大まかに以下です。

#### 人間の役割

- 要件・仕様の整理
- ドメインの設計
	- ドメインのロジックはソフトウェアとしての競争力の源泉なので、ここを弱いコンテキストで生成できる場合そもそもあまり作る価値がない、という気持ちで人間の業務としています
- coding agentへの業務の指示 / 成果の受け入れ
	- coding agentにはいいソフトウェアを作りたいという内発的な動機が無いので、人間の側に成果物に対する明確な信念がないとぼんやりした機能ができてしまいがちです
	- 業務の指示には、大まかにアプリケーションの実装とcoding agent自身のための環境の構築があります
		- 例えばスキルやドキュメントを書いたり、coding agentが読めるloggerの導入をしたり
- plan / コードのレビュー
	- 将来的には人間がやらなくなる部分だと思いますが、まだ見たい気持ちを捨てきれませんでした
	- 成果物の挙動だけを見てすべての仕様を察することはなかなか難しいので、しばらくは続けるつもりです
- 給与を受け取る

#### coding agentの役割

- アプリケーションの設計・実装・デプロイを行う
- 自身のための環境の整備を行う
- 情報を集約する
	- インフラのメトリクスやエラー、ある機能の仕様、スケジュールの確認など

この決定も、 `role.md` として保管しています。

## 実際の運用

上の方針を決めて実際に開発のスタイルを変更した際のtipsを列挙していきます。

- 今まではCursorのみを使っていたが、Claude Code + viewerとしてのCursorに変更
- 設計に関するドキュメントをすべて `/docs` に保管しておく
- 常に与える類のコンテキストを極限まで圧縮する
	- CLAUDE.mdには、1手目を見誤らない という観点の情報を5行くらい書いています
	- TypeScriptの場合、何も指示しなくても概ね癖のない高品質なコードが出てくる印象です
- PRを出すたびに作業のログと結果をまとめて保存しておく
- 各種スキルを用意する
	- 特定の手順で静的解析、ユニットテストを行うスキル
	- PRの作成を行うスキル
	- インフラのメトリクスを取得するスキル
	- などなど…
- Git操作やデプロイなどCLIで行える作業はすべてClaude Code経由で行うように心がける

## 現状どうか

うまくいっています。

あまり本質的な指標ではないですが、今まで1日に1-2PRを作成していたのが3-4に伸びています。  
コードの品質についても特に落ちている印象はないですが、これは既存の制約を受けにくい新規機能の実装をしていたからという側面も強いかもしれません。

敢えてマイナスを挙げるとすれば、エージェントが届かない情報（口頭でやりとりした内容、CLIからアクセスできないサービスなど）を伝達するというタスクを人間がやらなければいけないのですが、これは控えめに言っても退屈で、疲れるというくらいです。  
また、好みの部分ですがLLMの生成する文章は癖が強く網羅的で、要点だけを的確に伝えるPRを作成するのにはやや不向きなのかなと感じるときはあります。

今後の展望としては、非同期ジョブで常にcoding agentが動いてコードの品質担保を行っている、というようなアーキテクチャが組めるとよいなと思っています。
