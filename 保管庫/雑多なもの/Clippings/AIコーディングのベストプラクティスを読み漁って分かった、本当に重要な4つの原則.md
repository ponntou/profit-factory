## はじめに

ここ一年ほどで、AIを使ったプログラミングに関する記事が爆発的に増えました。OpenAIやAnthropicの公式ドキュメント、個人ブログ、Zennの記事など、様々な場所で「AIコーディングのベストプラクティス」が語られています。

ただ、記事が増えすぎて「結局何が一番重要なのか」「どの記事を読めばいいのか」が分からなくなってきたと感じている人も多いのではないでしょうか。私も競プロ・Kaggle・Web開発・個人開発など様々な場面でAIコーディングを実践してきましたが、記事ごとに主張が微妙に違っていたり、ツール固有の話と汎用的な話が混ざっていたりして、整理が必要だと感じていました。

この記事では、 **ツールに依存しないAIコーディング実践知** に焦点を当て、ネット上の記事を読み込んで4つの大きな原則にまとめます。

## AIコーディングの4つの重要原則

## 1\. プロンプト設計：AIに「何を・どう・なぜ」を伝える

AIに何かをやってもらう時、「とりあえず指示を投げる」だけでは期待通りの結果が返ってこないことが多いです。プロンプト設計は、AIに「何を・どう・なぜ」やってほしいかを明確に伝える技術であり、AIコーディングの基礎中の基礎です。

この話題を扱っている記事とtips一覧

**1\. Best practices for prompt engineering with the OpenAI API**

- 指示と入力を `###` や `"""` で区切って、AIが「何をすべきか」と「何に対して行うか」を混同しないようにする
- 出力フォーマットを例で示して、出力の形を固定する
- 制約は「〇〇するな」より「〇〇せよ」と肯定形で書く

**2\. Prompt engineering (OpenAI API docs)**

- 指示を「目的 / 制約 / 出力形式 / 例 / 参考情報（コードや仕様）」に **セクション分け** して書き、見出しや区切りで境界を明確にする
- Few-shot（入力→期待出力の例）を少数入れて、 **コードの粒度・スタイル・フォーマット** を誘導する
- 失敗例やチェックリストなどの **小さな評価セット** を用意して反復し、良い形をテンプレとして再利用する

**3\. Prompting best practices (Anthropic)**

- 期待する出力が「上振れ」なら、それを明示的に要求する（曖昧さを減らす）
- 制約に「なぜ重要か」（動機/理由）を添えると、判断が必要な場面で一貫性が上がる

**4\. How the Best People Actually Use AI**

- 曖昧な指示を避け、具体的に書いてAIの自由度を削る（具体例つき）
- コンテキストが劣化したら「重要情報だけをコピペして新規チャットで再開」する（copy-paste reset）

### 必要な情報を与える

多くの人にありがちなのが、「〇〇をして」とだけ指示して、必要な情報（関連するコード・ファイル・期待する実装方法や出力結果）を与えないパターンです。これはAIに実装方法を完全に任せてしまっているため、期待通りの出力を得られる確率が低く、出力も安定しません。

特に以下の二点を守ることが大切だと感じています：

1. 自分から与えないといけない情報はしっかり準備して与える
	- AIに代わりに作業を **やっていただく** ので、こちらも誠意を見せようという心構えが大切です
	- 例えばデバッグしてもらいたいなら、背景情報 (この部分を変更したら動かなくなった等)・現在の出力結果・期待される出力結果・該当箇所のコード・なぜ壊れているかの予想などを情報として与える
2. このコードを書く目的・背景は何かを共有する
	- これは、「何のためのコードなのか」によって期待されるアウトプットの種類が全く異なるからです。少し例を挙げると：
		- **既存コードのリファクタリング：** 他ファイルとフォーマットを統一してもらう必要がある
		- **プロトタイプをゼロから作る：** 複雑で「完璧な」コードではなく、今後改善がしやすいように明確なディレクトリ構成・モジュール分けを作ることに集中してもらいたい
		- **デバッグ：** コードを綺麗に「より良く」するのではなく、バグの原因を発見してそれに関連する部分だけを簡潔に直してもらいたい

### 出力フォーマットを例で示す

例えば「データの分析結果をJSON形式で出力する」「プログラムのドキュメントを書く」といったタスクにおいては、期待する出力フォーマットを具体的に例示してあげることが大切です。  
AIに「こういう形式で出力してほしい」と言葉で説明するより、実際の例を1〜2個見せる方が圧倒的に精度が上がります。

JSONであればキーの名前や階層構造、markdownドキュメントであれば主要なセクション名くらいは自分で考えておくと、一発で期待通りのアウトプットが得られます。

### 制約は「こうせよ」で明示する

*「余計なコメントを入れるな」* と否定形で書くより、 *「コメントは関数の上に1行だけ、目的を書く」* と肯定形で書く方が、AIは従いやすいです。

また、「なぜその制約が重要か」を添えると、判断が必要な場面での一貫性が上がります。例えば「パフォーマンスが重要なので、ループ内での文字列結合は避けてください」のように理由を書くと、AIは似た状況でも目的に則った適切な判断をしてくれます。

## 2\. 計画とタスク分割：Think first、いきなり実装させない

プロンプト設計ができても、複雑なタスクを一発で実装させようとすると失敗します。AIに「迷わせない」ためには、実装前に計画を立て、タスクを適切な粒度に分割することが重要です。

この話題を扱っている記事とtips一覧

**1\. The complete claude code tutorial**

- Think first: 実装前に設計・前提・ゴールをテキストで固める（Plan mode推奨）
- 「認証システムを作って」ではなく「email/password認証、Redisでセッション管理、/api/protected以下を保護」のように具体的に指示する

**2\. An example of LLM prompting for programming**

- Generated Knowledge: コード生成前に「実装方針を箇条書きで教えて」と聞き、計画を固めてから実装する
- 実装戦略を事前に与えることで、AIが迷わず一貫したコードを生成できる
- Master planを作り、「タスク1: 〇〇、タスク2: ××」のように番号で参照しながら分割実装する

**3\. Conversation: LLMs and the what/how loop**

- "what（意図）"と"how（実装）"を往復して設計を固める
- TDDでテスト（what）を先に書き、実装（how）に引きずられないようにする
- インターフェースが安定するまで、whatとhowを反復して抽象化を調整する

**4\. 実践フルAIコーディング**

- 設計段階で作業見積もりをさせ、編集対象（ファイル名/行番号）やI/Oなどを明示させる
- テスト変更と実装変更の分離など、フェイズ分割を前提に進める

### Think first: 実装前に計画を固める

複雑なタスクでは、いきなり実装させるのではなく、まず計画をテキストベースでまとめさせると一発成功率が圧倒的に高くなります

実装を複数段階に分けることでLLMの出力の質が向上するというのは、本当にたくさんの記事で言及されています。コード生成前に実装計画をAI自身に生成させ、それを確認・修正してから実装に進むことで、より良い最終成果物が得られます。

### 「後で修正すれば良い」が良い場合、良くない場合

別に間違っても後で修正すれば良い、という考え方もあると思います。実際、細かい調整やUIの変更などは最初から決めすぎるよりも後で変更して行った方が良いことも多いです。

しかし、「そもそもどのパッケージ/手法/アルゴリズムを使うか」や「全体のファイル構成をどのようにするか」のようなより根本的な部分が期待通りでない場合、修正するくらいなら一から作り直した方が良い、という状況になってしまいます。

貴重なAPIトークンを無駄にしないためにも、「この部分についてはもし間違っていても後で修正すればいいから今の時点ではあまり細かく指定しなくてもいい」「この部分は実装プランの時点でAIと認識をすり合わせておかないと面倒臭い」のように、指示しなければいけない・指示しなくても良い内容を見極めることも大切だと感じています。

後で修正すれば良いタスクについては、下の反復ループの考え方が重要になってきます。

## 3\. 反復ループ：一発で当てない。直しながら当てにいく

AIコーディングで最も重要なのは、「一度で完璧なコードを生成させよう」という考えを捨てることです。最初の実行では調整しやすい基盤となる構造を作り、細かい部分は後で修正していく方が、結果的に速く高品質なコードが得られます。

この話題を扱っている記事とtips一覧

**1\. Prompt engineering overview (Anthropic)**

- プロンプトを「一発で当てるもの」ではなく「評価しながら育てるもの」として扱う
- 成功率を測りながら、効果の大きい改善から順に適用する

**2\. Your job is to deliver code you have proven to work**

- 手動検証＋自動テストの両方を行う
- エッジケースを試して壊す

**3\. LLMのプロンプト技術まとめ**

- Self-Refine（自己フィードバックで反復改善）/Self-Consistency（複数回推論して多数決）など、反復で精度を上げる手法を整理している

### 最初から完璧を目指さない

多くの人にありがちなのが、「一回の実行で完璧なコードを生成させよう」として、最初のプロンプトに全ての要件を詰め込んでしまうパターンです。これは逆効果で、AIが迷走したり、トークンを無駄に消費したりする原因になります。

**効率的なアプローチ：**

1. **最初の実行** ：調整しやすいベースの構造を作ることに集中する
	- 主要な関数・クラスの骨組み
	- 基本的なロジックの流れ
	- ファイル構成とモジュール分け
2. **後の実行** ：細かい部分を段階的に改善する
	- エラーハンドリングの追加
	- エッジケースへの対応
	- UIの調整やパフォーマンス最適化

### テストで自己修正ループを回す

AIに生成されたコードをテストする方法を与えることで、自動的に品質を改善するループを作れます。これはテスト駆動開発（TDD）の考え方をAIコーディングに応用したものです。  
複雑なアルゴリズムが含まれているようなプログラムを書く際に特に有効なテクニックです。

**基本的な流れ：**

1. **テストを書く（または書いてもらう）**  
	プログラムの出力結果を書き出し (ログ)、デバッグに必要な情報をすべてtxtファイルなどに出力できるようにする。また、web開発などで明確に通したいテストケースなどがあれば、それを実行するコマンドをAIに教える。
2. **AIに実装させる**  
	*「プログラムの出力結果を随時確認しながら、〇〇という挙動になるまで改善ループを回しながら実装してください」* *「このテストが通るように実装してください」* と指示する。最大テスト実行回数なども一緒に伝えることで、実装に詰まった場合に無限ループになってしまうのを防ぐ。
3. **反復する**  
	AIの実行後、自分でも一応テストしてみる。期待通りだったらOK。もしバグを見つけた場合は、出力ログのどの部分が期待通りでないかを具体的にAIに伝えて、もう一度デバッグしてもらう。  
	もし出力結果にエラーやバグがあった場合、同じチャット内でもう一度デバッグしてもらうとまた同じ問題で止まってしまう可能性が高いので、別チャットに移動する・エラー原因の予測など追加情報を与えるなどの工夫が必要です。個人的には、3回試しても直らなかったら諦めて自分でデバッグするしかないと思います。

AIにテストケースを書いてもらう場合には、実装をするAIとはコンテキストを分けることが重要です。これは「テストを通すために、実装ではなくテストそのものを変更する」という挙動を防ぐためです。テストはテストで、実装は実装でというように、それぞれを批評的な立ち位置に置くのが良いです。

また、実装AIが通らないエッジケースのテストを通すために「本筋のロジックとは関係ないif文を挿入する」といった挙動をすることも考えられます。これは静的なテストケースがある限り完全には防ぎ用がないので、プロンプトでそういった挙動を禁止しつつ、 **プロパティベーステスト** など性質の違ったテスト方法を探索することも大切です。

## 4\. コンテキスト運用：会話は長いと壊れる。メモと役割分担をする

会話が長くなるほど、AIは最初の指示や重要な制約を忘れやすくなります。これは避けられない現象なので、「会話を分ける」「重要情報を外部に保存する」ことを前提に運用する必要があります。

この話題を扱っている記事とtips一覧

**1\. The complete claude code tutorial**

- コンテキストはトークン使用量20〜40%あたりから品質が落ち始める前提で運用する
- 1チャット1機能に絞り、タスクをスコープで分割する
- 進捗や計画は外部ファイルに保存し、次のチャットで読み込む

**2\. How the Best People Actually Use AI**

- コンテキストが劣化したら、重要情報だけをコピペして新チャットで再開する
- 会話を「ステートレス」として扱い、毎回必要な情報を明示的に渡す

**4\. 実践フルAIコーディング**

- AGENTS.mdの例を載せ、絶対ルールや報告形式などを外部化する
- 「そのセッションは汚染されているので捨てる」を鉄則として挙げている

### 会話が長くなるとアウトプットの質が落ちる

会話の長さ（トークン数）が増えるほど、AIは最初の指示や重要な制約を忘れやすくなります。以下のような状態になったら早めに違うチャットへ移る (コンテキストをリセットする) 方が良いです：

- 最初に指定した制約（「コメントは書かない」「この関数は変更しない」など）を無視し始める
- 同じような質問を繰り返す
- 最初の方に実装した内容と矛盾するコードを生成し始める
- その他明らかに出力の品質が落ちる

基本的に「1チャット1テーマで分業する」を徹底するのが良いアイデアだと思います。例えば、実装用・デバッグ用・質問を聞く用など、こまめにコンテキストを分けることが大切です。

### チャット間での情報共有の仕方

複数のチャットを併用する場合、「これまでの進捗・現在の状況」や「守らなければいけない制約」を次のチャットに引き継ぐ必要があります。これを手動でやるのは面倒なので、主に二つの方法があります：

1. **共通する情報はテキストファイルとして保存しておく** ：
	- プログラムの概要・制約・以前リサーチした関連手法の内容など、すべてのチャットに共通する内容については、それぞれのプロンプト内で個別に説明するのではなく、外部のファイルに保存しておいた上でこれをチャットに貼り付けるという方法が楽です
	- 長い実装を複数のチャットに分けて頼む場合は、TODOリストのような進捗を外部ファイルに書いて随時更新させることで、別チャットに移っても現在の進捗を共有できます。エディタによってはPlanモードで実装プランのマークダウンを作ってくれるものもあります
2. **チャットの最後で次のチャットへのプロンプトを生成してもらう** ：
	- これは私も個人的に良くやるのですが、会話が長くなってきたら、 *「違うチャットで〇〇について議論したいので、プログラムの概要・現在までの進捗・問題点などについてもまとめた上で次のAIへのプロンプトを数段落で作成して」* と頼んで次のプロンプトを作成してもらい、それをコピペするという方法です

## まとめ

この記事では、AIコーディングの実践知を4つの原則にまとめました。

1. **プロンプト設計**: 必要な情報を与え、出力フォーマットを例で示し、制約を肯定形で明示する
2. **計画とタスク分割**: Think firstで実装前に計画を固め、根本的な部分は最初に決める
3. **反復ループ**: 最初から完璧を目指さず、テストで自己修正ループを回す
4. **コンテキスト運用**: 1チャット1テーマで分業し、チャット間で情報を適切に共有する

改めて、AIの登場によってプログラマーに求められるスキルの種類が変わったなと感じています。細かい構文の暗記や定型的なコードの記述といった「手を動かす部分」に加えて、抽象的なレベルでの設計力、問題を言語化する力、品質を評価する力、テストを設計する力といった「何を作るか・どう作るか」を考える部分が、以前にも増して重要になっていると思います。
