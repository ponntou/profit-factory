## はじめに

こんにちは、TOKIUMの重村です。25卒として入社した後、新卒未経験でエンジニアになり、現在はAI出張手配プロダクトの開発チームリーダーをしています。

若手エンジニアの皆さん、Claude Code Skills使っていますでしょうか？

少し前からSkillsが登場し、できることの幅がすごく広がりましたよね。自分も早くからキャッチアップしていきたかったのですが、なかなか手をつけられずズルズルと学習しないまま来てしまいました。気づけばどんどん進化していき、Xの人達の投稿をみると「ここまで使いこなしているのか...」と焦りを感じています。

今回の記事では、そもそもなぜ革新的な技術に手をつけられなかったのか？実際にSkillsを設定してみて感じたこと、そして **失敗から学んだ「目的を明確にすることの重要性」** を書きました。同じように技術トレンドに対しての焦りを抱えている若手エンジニアの方には、ぜひ読んでいただきたいです。

## とりあえずSkillsを作ってみたが...

TOKIUMでは「AI 100K」という制度があり、月10万円までならAIツールが使い放題となります。（本当に恵まれた環境ですね）

そんな恩恵を受けながら、僕の使い方はかなり雑で、「このエラー何？」と聞いてみたり、頑張ってもJira MCPを使ってチケットから実装方針を考えさせたりと、使いこなせている感覚は全くありませんでした。

このままではもったいない！と思いXやZennの記事を見漁っては保存して、「これを元に色々作ってみよう」と、Claudeと一緒に色々作ってみました。

「TypeScript + Hono + Zodのスキル」「特定の観点からレビューをするスラッシュコマンド」「セキュリティチェックをするサブエージェント」etc…

設定は一応できました。 **でも、業務で使うイメージが全く湧きませんでした。**

普通のClaude Codeを使っても、言語やライブラリの知識が足りないとはあまり思ったことはない。セキュリティチェックといっても、自分がセキュリティの知識を持っていないので、正しいレビューをしているのかよくわからない...。

今振り返ると、実務での開発効率性をあげることが目的ではなく、 **「Skillsを作ること」が目的になってしまっていた** んです。

## どんなSkillsを作ればいいのか

ちょうど同じ時期、チームの先輩エンジニアが別プロジェクトに異動することになりました。この先輩エンジニアには、本当にお世話になっていました。

- ペアプロでのアーキテクチャレビュー
- 命名について議論
- LLMアプリの設計でどこまでLLMに任せるべきか、の相談

など、色々と教えていただきました。先輩がいなくなると思うと「これを全部、一人でやるのか...」と、不安が押し寄せてきました。と同時に、ふと思いました。

**「ペアプロでやっていたことをSkills化して、Claude Codeにやってもらえばいいんじゃないか？」**

Skillsをとりあえず作るのではなく、「先輩に聞いていた業務を効率化したい」という具体的な目的。ここから逆算すれば、どんなSkillsを作るべきか見えてくる気がしました。

とりあえず、先輩によく相談していた「LLMアプリの設計アーキテクチャ」のSkillsを作ってみました。処理をLLMに任せるべきか、それともロジックにするべきか、場合によっては自分だけで判断するのが難しく、Skillsにもってこいだなと思いました。

以下が実際に作成したSkillです（※省略版）。

```
name: llm-app-architect
description: |
  LLMアプリケーション実装の専門家。LLM介入を最小化し、決定論的処理を最大化する設計を行う。
  トリガー例:
  「LLMで〜を実装したい」「LLM使って〜作りたい」

### 核心原則

> LLMでなければできない処理にのみLLMを使う

[自然言語入力] → [LLM: 構造化] → [API/DB] → [決定論的変換] → [LLM: 柔軟な応答生成]
                  ↑必要な場合のみ                               ↑必要な場合のみ

### なぜLLM介入を最小化するのか

| 観点 | 課題 |
|------|------|
| 再現性 | 同じ入力に対して同じ出力が保証されない |
| テスタビリティ | 出力が非決定的なため、従来のテスト手法が使いにくい |
| コスト | API呼び出しごとに課金される |
| レイテンシ | 1回の呼び出しで数百ms〜数秒かかる |
| デバッガビリティ | 問題発生時の原因特定が困難 |

### 設計パターン

**パターン1: 入口LLM（最も推奨）**

ユーザー入力 → LLM → 構造化データ → 外部API → 結果

**パターン2: 出口LLM（条件付き許可）**

データ → LLM → 自然言語出力

※テンプレートでは表現できない柔軟な生成が必要な場合のみ

**パターン3: サンドイッチ型**

入力 → LLM(構造化) → 決定論的処理 → LLM(応答生成) → 出力

### LLM介入の判断フロー

1. ルール/テンプレートで表現できる？ → **Yes** → 決定論的処理
2. 自然言語の理解・生成が必要？ → **Yes** → LLMを使う
3. 曖昧な判断だがパターンが有限？ → **Yes** → ルールベースで対応
4. 曖昧でパターンが無限？ → LLMを使う（スキーマで出力を制約）

### 設計の原則

- LLM出力は必ず構造化出力のスキーマで厳密に定義する
- 中間処理は決定論的に保つ
- フォールバック（デフォルト値、バリデーションエラー処理）を定義する

（以下、設計ワークフロー、チェックリスト、出力形式、アンチパターン等は省略）
```

革新的だったのは、 **先輩の時間を使わずに同じような議論がClaudeとできたこと** でした。

今まではこういった設計思想が僕と先輩のみの暗黙知として共有されていたため、Claudeと同じような議論をするには詳細な前提知識を伝えなければならず、安定したアウトプットを出させることは極めて難しかったです。

しかし、Skillでこの前提知識をインプットさせたことにより、わざわざセッション毎にやりとりをしなくても、最初から「LLM介入を最小化する」という設計思想を共有した状態で議論を始められるようになりました。

目的が明確だったからこそ、Skillsの方向性も明確になりました。「先輩とのペアプロを再現する」というゴールがあれば、SkillCreatorを使って実務に使えるSkillsが簡単に作れると感じました。

## 作って終わったSkillsと、使われるSkills、何が違ったのか

2つの体験を振り返って、失敗と成功の違いを整理してみました。

新しい技術にどんどん触れることはもちろん大切ですが、使いこなし、開発生産性を向上するためには **「何を解決したいか」が先。「どんなSkillsを作るか」はその後** だと考えています。

SNSで使いこなしている人を見ていると、「この設定で開発生産性が爆増した」などの投稿をみると、ついつい丸ごとコピペしちゃおうと思っていました。

しかしプロジェクト毎に前提や必要な知識・思考は必ず違うはずであり、コピペするだけでは実用性のあるものにはなりません。大切なのは **「何を解決したいか」を明確にすること** だと感じました。

## 焦って技術トレンドに飛び込む前に、やってほしいこと

そもそも新卒未経験の自分にとっては学ぶべき技術は山ほどあります。基礎的なプログラミング、設計パターン、チームでの開発フロー...。なのにさらに新しい技術が次々と出てくる。技術トレンドの洪水に溺れそうになっていました。

同じように情報の波に溺れそうになりながら焦っている方がいらしたら、まずは以下のステップをやってみてほしいです。

1. **実務や学習で詰まったことを書き出す**
2. **先輩に聞いたことをリストアップする**
3. **先輩がどう教えてくれたから解決できたのか、学べたのかを思い出す**

ここから自分が本当に効率化したい業務が見えてくるはずです。業務効率化が目的なら、SkillsやAIを使うことを目的化しないでほしいです。

自分の学びたいこと、取り組むことを達成するために、Skillsを「道具」として使いましょう。焦って新技術に飛びつく前に、立ち止まって分解してみてほしいです。

## 実際に作成したSkillsの例

上記で紹介したLLMアプリケーションのSkills以外にも、以下のようなものをSkill化してみました。

- **命名規則・命名レビュー** ：変数名、関数名、クラス名の適切な命名を判断するSkill
- **コードレビュー観点** ：PRレビュー時のチェックポイントをまとめたSkill
- **テスト設計** ：単体テスト・結合テストの境界、テストケースの作り方をまとめたSkill

どれも若手エンジニアが一度は悩むポイントではないでしょうか。先輩に聞けば一瞬で解決するけど、毎回聞くのも申し訳ない…そんな知識や暗黙知をSkill化することで、Claudeがいつでも先輩代わりになってくれます。

今までは先輩と会話して初めて気づくことも、実装時やPR作成前に自分で気付けるようになりました。

## おわりに

どんどん技術が進歩していくこのご時世に、焦る気持ちは本当によく分かります。Xを開くたびに自分が遅れているのではと焦り、追いつかねばという使命感に駆られます。

ただ、追いつくだけでなく、 **真の意味で自分のものにする** ためにも実際に役立つものを作るステップを挟んで欲しいなと思います。

まずSkillsで効率化したい業務は何か、しっかり見極めてほしいです。そこに効くSkillを作ってほしいです。

そうしてこそAIを本当の意味で使いこなせるようになると、私は考えています。
