## そこのPMさん。「AIを使いこなしてる」って胸張って言えるかい？

現在所属している医療システムのチームでは、  
PDMやドメインエキスパートもエンジニアも、  
AIを活用して設計・要件定義や調査を行っています。

ただ、PDMやドメインエキスパートの方々は、それでもまだAIを  
「フル活用できていない」という悩みを抱えているそう。  
(おそらく他の会社に比べたら圧倒的に使い倒しているが)

GeminiやCursor、ChatGPTのチャット画面は活用するものの、  
特にカスタマイズせずに素の状態で使っていたり。  
また、各所に点在している会話やドキュメントを集約できないまま、  
バラバラに管理している状況も悩みのタネのようでした😭

前の現場でも、スプシで管理していたりNotionやGitHubや、  
色んなとこに情報が点在している。

よくあるのは、Slackで議論し、Notionにも書かれていて、  
でもリポジトリの中身もわからないと実現イメージが湧きません。  
みたいなケースもあり、ツールの往復で頭がパンパンに🤯

加えて、情報が時系列で流れてしまうことで  
「あれどこやったっけ」って内容を  
すぐに参照できないことも個人的な課題でした。

## 「チーム開発向けに活用し切れてない」のはもう遅い

というのが正直な感じかと思います。

また、これを書くとちょっと偏見があるように見えるかもしれないですが、  
非エンジニアにとってAI = サルにとっての火

そういう意味で🔥(AI)でより具体的などんな使い方ができそうか？  
PM、PDMレイヤーの人にもハンズオンする価値はありそう。

## とりま、こだわりがないならCursor入れとこ？

まずやるべきは  
システムのコードを直接参照できるチャット画面を  
ローカルのPC環境に用意すること。

エンジニアはいろんなエディタ、いまやターミナル(黒いコマンドしか打てない画面)  
しか使わんって人もいます。  
しかし、PM,PDMであればそんな拘る必要もなく、  
まずはCursorというエディタから入るのが一番とっつきやすいでしょう。  
GeminiやChatGPTのチャットでも出来る人はいますが、  
システム開発に関わっているのであれば、本番コードを  
直接AIが参照できる環境で作業したほうが確実です。

理由はシンプルで、毎回必要な情報をコピペしなくていいからです。  
チャットでトラブルシューティングの依頼をCursorのエディタで依頼すれば、  
勝手にコードを調査して当たりをつけてきます。

もう、あなたが最初から調べたり推測する必要もなくなります。😄  
それでも解決しないときにはあなたの出番です。

無料プランもありますが、これを機にAIの活用力を極めたいなら20ドルプランを契約が吉。

## まずはMCPを連携する

ただチャットでやり取りするだけのAI活用から一歩進み、  
横断的にドキュメントや会話履歴を集約しましょう。  
そうすることで「意思決定のための情報を一箇所にまとめる」  
または「意思決定に必要なソース元はどこか？」を  
容易に管理する方法を紹介します。

MCPとはなんぞや、を説明すると長くなってしまうので、  
要点だけかいつまんで一言で言うと、  
「SaaS画面を見なくても、認証したらSaaSの情報を取れる直通トンネル」  
ドラえもんで言う「とおりぬけフープ」です。

そうです。あなたはまず、  
「いろんなSaaS画面を往復すること止める」  
ということからスタートするのです。

### SlackのMCPを連携する

Slackの連携設定については、エンジニアに確認してみてください。

SlackのBotの設定が必要になるため、難しい場合、  
トークンの発行はエンジニアまたは情シスに依頼してください🙏

設定手順についてはこちらの記事も参照してください。

この連携をすることで、特定の会話スレッドの中身をリンク1行読み込ませるだけで  
要約や必要な情報を抜き出してくれます。  
また、Cursorで壁打ちした内容をそのままSlackに投稿してチーム内共有もできます。

例えば、Notionで分析や調査結果のIssueを作成したことをそのままツール横断せず、  
Cursorの画面上から  
「@〇〇さん [https://notion.so/](https://notion.so/)....のIssue作ったので確認お願いします」  
とSlackに投稿できます。

例:  
slackのメッセージのリンクを共有し、会話スレッドも含めて要約や、やるべき事を洗い出し  
![](https://storage.googleapis.com/zenn-user-upload/6b9e48d8b8fc-20260309.png)  
※ 「slackのMCPを使って」「use slack mcp」と書いてあげたほうがいいです。  
じゃないと、勘違いして直接slackのURLを検索しに行ってエラーになることあるので。

### NotionのMCPを活用する

会社でNotionを使ってドキュメントやナレッジ管理をしているなら、  
それを連携しない手はありません。

Notion AI単体で使うよりも、Notion MCPや  
SlackのMCPと組み合わせることで効果が高まります。  
具体的には精度の高い仕様・ナレッジの集約が実現できます。

公式の設定方法はこちらです。

SlackとNotionのMCPを連携することで、  
情報の取得だけでなく、情報の出力も可能になります。

SlackやNotionおよびリポジトリの内容を把握した上で、  
その内容をNotionのページとして出力できます。  
この意思決定は

- このSlackのこのリンク
- このNotionのこのページ  
	をもとに行い、このようなアクションプランを立てた  
	という共有を  
	外部のメンバーやビジネスサイド・経営層に向けて見やすいフォーマットで出力できます。

### その他のツールへの応用

ここではNotionを例に挙げましたが、JiraやAsana、BacklogのMCPもあるはずなので、  
チーム環境に合わせたカスタムが可能です。

Jiraの設定はこちらを参考にしてください。

### Cursorでの設定方法

Cursorで設定する際は、Cursor Settings の「Tools & MCP」から  
「New MCP Server」（または「Add Custom MCP」）を選択します。  
![](https://storage.googleapis.com/zenn-user-upload/9435718760c0-20260309.png)

以下のように設定してください（ `DUMMY` の部分は必要なトークンに差し替えてください）。

```
{
  "mcpServers": {
    "notion": {
      "url": "https://mcp.notion.com/mcp"
    },
    "slack": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-slack"],
      "env": {
        "SLACK_BOT_TOKEN": "DUMMY",
        "SLACK_TEAM_ID": "DUMMY"
      }
    }
  }
}
```

おそらくNotionとか認証が必要なものはこんな感じでConnectボタンが出るのでクリック  
![](https://storage.googleapis.com/zenn-user-upload/40ea729aa136-20260309.png)  
これで設定自体は完了です。（必要に応じて、Notionの認証が求められる場合があります）

## Redashのクエリを作成する

SQLが書ける方であればSQL構文のイメージはつくと思います。

しかし、エンジニアほど実際のデータベースの構成に詳しい方は少ないと思うので  
いざ必要なデータを取りたいと思ったときに「〇〇のデータってどこに何のカラムで保存してたっけ？」  
というエンジニアとのコミュニケーションコストが発生します。

中にはレスが遅い人もいたりすると、その間に他のタスクやって  
返信が来て、えーっと、なんだっけか？という感じの頭の切り替えする。  
地味に面倒ですよね。

なら、Cursorくんに調査してもらっちゃいましょう。

ポイントは、リポジトリ内のスキーマをCursorに読み込ませることです。  
カラムやJOINの関係・親子関係を把握した状態でクエリを生成できるため、  
エンジニアへの逐一確認が不要になります。

AIに「どの項目を取ってくればいいか」を聞くだけで済み、  
分析・調査のタスクをより非同期的に進められます。

**3ステップで始める**

1. テーブルの定義があるフォルダ or ファイルの相対パスをコピーする ※わからない場合は事前にエンジニアに確認

以下のキャプチャのように見られるはず  
![](https://storage.googleapis.com/zenn-user-upload/b12b45a0e64b-20260309.png)  
2\. 以下のプロンプトで定義内容を読み込ませる

```
packages/db/src/schema
このファイルを読み込んでデータベースの定義を把握してください。
```

正しく依頼できればこんな感じで読み取ってくれるはず  
![](https://storage.googleapis.com/zenn-user-upload/9538b57ebba9-20260309.png)  
3\. 書きたいSQLのイメージを依頼する

```
今読み込んだ内容をベースに、TODOを登録してから3日以上アクションがないユーザーのSQL文を作ってください。ファイルは作らず、コンソールに出力してください。
```

こんな感じで生成してくれるはず  
![](https://storage.googleapis.com/zenn-user-upload/1fdc3c237d86-20260309.png)

> ファイルは作らず、コンソールに出力して

この一文はなくてもよいですが、AIが意図せずファイルを作成することもあるため、  
リポジトリをきれいに保つために添えておくことをおすすめします。

## プロンプトのコツ

プロンプトに以下の考慮を加えることで、出力の精度が上がり、  
ハルシネーション（もっともらしい嘘をつくこと）が少なくなります。

**1\. 「わからないことは聞いて」と指示する**

これだけで効果が出ます。

AIは基本的に不明な部分を推測で埋める性質があります。  
なぜそんなことをするかの説明は省きますが、そのため  
「不明点はユーザーに確認すること」を明示することで、  
コンテキストを勝手に補完せず確認を取るフローを作れます。

その結果、AIには足りない暗黙の前提が可視化され、  
自分の中にしかない情報をアウトプットしやすくなります。

**2\. エビデンス・リソースを必ず提示すること**

ハルシネーションが起きた際、それが誤りかどうかを判断する根拠がないと困ります。

指示する際のプロンプトに  
「判断に至った証拠やエビデンスは必ず明示すること」  
と指示することを忘れないようにしましょう。

こうすると、「この結論はNotionのこのページから判断しました」のように  
リソースを紐づけて出力するようになります。  
誤りがあった場合もユーザー側から正しい情報を提示しやすくなります。

例を上げましょう。  
以下のような背景です。

- インフラを新規サービスは切り分けたい
- 基本はSREが対応するが、速度優先で開発チームで対応したい
- 計画書はNotion, SREとはSlackでテキストコミュニケーション。

(実際にチームで発生したケースを参考にしています。)

このようなときに、PDMが今のステータスや、次に誰に何を確認するべき？  
どんなことを先に立ち回っておくとエンジニアがスムーズに行ける？

ということを把握したいとしましょう。

その場合に以下のようなプロンプトを用意して  
Cursorに依頼してみましょう。

```
https://www.notion.so/... 
https://XXX.slack.com/archives/.... 

slack mcpとnotion mcpを使って読み込んでください。

ここでインフラを既存リソースから独立させる計画とSREとの議論を行いました。

これによる
決定事項
懸念事項
アクションプラン

を整理してください。

不明点があればユーザーに「箇条書き」で確認してください。
```

するとこんな感じで整理されます。  
![](https://storage.googleapis.com/zenn-user-upload/5ecb1b67da12-20260309.png)

その後に、こんな感じで、あなたに追加で聞きたいことを尋ねてくれます。  
![](https://storage.googleapis.com/zenn-user-upload/75914fda8e58-20260309.png)

そしたら、あなたはその対話をチャット欄で書き込むことで、  
その内容を考慮したアウトプットを出してくれます。

また、これを実際に以下のように会話しましょう。

人名まで入れて明確にしたい  
Notionに連携したいが、とりあえず情報を取捨選択したいので、mdファイルに出力して

そうすると、以下のようにmdファイルを作ってくれます。

## 懸念事項

### SRE 側の懸念（Bさん）

1. **SRE 専任不在による品質リスク**
   - Embedded SRE がいないため、指標管理や運用品質の担保が難しい
   - 「負担軽減より品質維持」を優先すべき。負担が増えるなら人員増強を検討すべき

2. **アカウント分離時のセキュリティ監査コスト**
   - アカウントを分離する場合、セキュリティ要件を満たすための**固定費が別途発生**する
   - 必要な要件を洗い出した上で、予算として許容できるか事業部と判断が必要

3. **アーキテクチャ懸念が多数残存**
   - 既存サービスと技術スタックの差分が大きく、全社標準からの乖離がある
   - 現状の構成になった経緯と運用体制・事業規模を踏まえた継続議論が必要

   | 観点 | 既存サービス | 新規PJT（計画） |
   |---|---|---|
   | コンテナ実行基盤 | サービスごとに独立 | 共有クラスタ |
   | バッチ実行 | 常駐型 | オンデマンド型 |
   | デプロイ手段 | 全社標準ツール | 独自ツール |
   | キャッシュ認証 | 標準構成 | 認証強化あり |
   | シークレット管理 | 環境変数 | 専用管理サービス |

4. **本番環境の運用品質担保**
   - SRE 専任不在の状態で本番運用の品質・セキュリティを担保できるか検討が必要

### 新規PJT Dev 側の懸念（Aさん）

5. **スケジュールがタイト**
   - 新規PJT のリリーススケジュールが固定されており、体制整備を待てない状況
   - Dさんの参入を待ちたいが、スケジュール上難しいのが実情

## 機能相談・トラブルシューティングはセクション分けで管理する

CursorやClaude、Codexなどエージェントツールにはエンジニア向けのPlanモードという概念があります。  
おそらくエンジニアはこれを使ってタスク分解してから実装に入る  
という方も多いのではないでしょうか？

うちのチーム内ではこれをさらに詳細化し、  
以下のフェーズ毎にファイルを分割する仕組みを作って管理しています。

- 1: specify（仕様定義）
- 2: plan（技術計画）
- 3: tasks（タスク分解）
- 4: implement（実装）※エンジニアのフェーズ
- 5: knowledge（ナレッジ更新）※エンジニアのフェーズ

PMやPOはフェーズ1〜3の計画書をAIと一緒に作ることを目標にすると良いでしょう。

よくある課題として、「なぜこのタスクをしないといけないのか」を口頭で話したが  
後で思い出せない、というケースがあります。

これを防ぐため、フェーズ1の段階で以下を明示するようにします。

- なぜこの機能をつけるべきなのか
- トラブルシューティングの発生原因
- どういった条件で発生するのか

こうすることで、実装フェーズでのエンジニアとPMの認識のズレが少なくなり、  
手戻りが減ります。

仕様書一枚で認識が揃えば、  
実装が正しいかどうかの確認も仕様書との照合だけで済みます。

また、このようなドキュメントをリポジトリ内に残しておくことで別のメリットもあります。

Cursorのリポジトリ検索機能（またはエディターの検索機能）で  
「なぜこの実装を行ったか」をすぐに探せるようになります。

SlackのスレッドURLやNotionのページURLをこのドキュメント内に残しておけば、  
後でエンジニアが情報の所在を探す手間が省けます。  
「このドキュメントを見てください」で済むため、コミュニケーションコストが下がります。

PMの稼働が取られることも少なくなり、  
チーム全体のフリーな作業時間の確保にもつながります。

コチラも例を上げましょう。

CSから不具合報告が...  
どうやら管理画面の特定の顧客管理ページが404が出て開けないとのこと。  
なんとなくslackのやり取りから不具合の原因は特定できそう。  
社内の管理画面なので優先度は低いが、修正しないと。

しかし、エンジニアは眼の前の大きめのタスクで手一杯  
ちょっとしたことであれば手伝いたいが、、、

みたいなときに、こんな感じのプロンプトを。  
段階ごとに整理してみましょう。

```
https://XXX.slack.com/archives/.... 

slack mcpを使って読み込んでください。

議論されている内容からどのような事象が起きて、どのような不利益が発生しているか
どのような状態であるべきか？
phase1 specify(要件定義)として出力してください。

出力先は docs/specs/[YYYYMMDD-{unique-num}]/spec.md

不明点があればユーザーに「箇条書き」で確認してください。
```

議論後  
↓

```
phase1の内容をもとに
既存のコードを調査して、原因の特定、該当する可能性のある箇所を洗い出してください。
そして、その上で解決策を検討したうえで

phase2 plan(技術計画)を出力してください。

出力先は docs/specs/[YYYYMMDD-{unique-num}]/plan.md

実装計画を立てるが、以下を必ず計画に組み込むこと
- 後方互換は本当に必要でない限り、一切組み込んではいけない。
- パッチ的な対応は一切NG、本質的な実装設計にすること。
- 有るべき最小の実装で済ませること。
```

その後  
↓

```
phase2 までの内容をもとに

phase3 タスク分解(小さく独立、分解可能)を出力してください。

以下を必ず計画に組み込むこと
- 実装規約のドキュメントがある場合はphase0で読み込ませるタスクを追加する
- 実装完了後、品質チェック/テストのコマンドを実行し、通過する確認をする
```

ここまでできたら、  
一旦これをタスク管理ツールで投げるか、PR出しちゃいましょうｗ  
そのままエンジニアが内容を把握し、レビューしてくれるはずです。

問題なければ、そのままあなたがCursorで

```
docs/specs/[YYYYMMDD-{unique-num}]/ の中身を見たうえで、
修正を開始してください。
```

と依頼したら、あら不思議。  
エンジニアじゃないのに修正ができちゃいました😇

と似たようなことが、別のチームで起きていましたｗ  
(PDMが不具合調査〜PR作成、マージまでｗ)

このphase単位の管理はより詳細に定義することで  
更に汎用的、かつ再現性の高いものになりますが、

ここではあくまでこのような概念で進める手があるよ、  
という方法論だけ共有するにとどめておきます。

## AIに流されるんじゃなくてAIを操れ。そして審美眼と言葉を磨け。

こういった具体的な例はいくつか書かせていただきましたが、  
それと同時に、あなたが気づかないといけないことがあります。

それはAIを使いこなせるかは

- あなたの言語化する能力に比例すること
- ハルシネーションをハルシネーションであると見抜けるか(ひろゆき風)
- あなた自身が何をゴールとしているか？が見えているか？

まずはこの3つを鍛えていく必要があること。  
これは人間とのコミュニケーションでも大事ですよね。

「言葉使えよ。人間だろ。」

ある新聞社のCMで猫がうぜーしか言えない子供にこのように語るシーンがあります。

あなたは猫でもサルでもなく、人間です。  
多種多様な言葉や意思疎通が行える、  
かつ言語能力も成長できる優れた存在なのです。

つまり、あなたがAIとの対話が上手になっていくということは  
すなわち人間とのコミュニケーションでもプラスに働くということです。

近々この記事をもとに、コミュニティ内で勉強会を開催しようと考えています。

記事を書くだけでなく、話す機会を作ることで、  
自分自身もこの活用法をブラッシュアップしていければと思います。
